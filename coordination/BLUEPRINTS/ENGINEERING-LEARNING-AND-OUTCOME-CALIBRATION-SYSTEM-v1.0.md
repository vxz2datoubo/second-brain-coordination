# 工程学习与结果校准系统 v1.0

> `agent_id: GPT`
>
> 定位：任务发布前预演、执行中观测、交付后校准、经验回写的项目级学习基础设施。
>
> 边界：默认 `research_only / NO_TRADE`。本系统不自行授权高风险操作。

## 一、核心目标

让每次任务都产生两类成果：

1. **直接成果**：代码、蓝图、报告、测试、数据合同或运行结果；
2. **学习成果**：哪些假设成立、哪些风险被低估、哪些做法带来额外收益、哪些控制失效，以及下次如何做得更好。

系统必须形成闭环：

```text
任务意图
→ 第一性原理拆解
→ 收益与负面影响预判
→ 风险门禁与用户告知
→ 执行和证据采集
→ 交付验收
→ 预测与实际对比
→ 正反馈强化 / 负反馈纠偏
→ 模板、规则、蓝图和任务路由更新
→ 下一任务使用更新后的经验
```

## 二、第一性原理任务拆解

发布每个非 trivial 任务前，GPT必须回答：

1. `fundamental_goal`：最终要改变什么真实状态；
2. `causal_mechanism`：为什么这些动作可能导致目标实现；
3. `constraints`：权限、时间、资金、数据、制度、安全和不可逆约束；
4. `required_evidence`：什么证据才能证明任务真的有效；
5. `failure_modes`：最可能怎样失败；
6. `observability`：哪些结果可直接观察，哪些只能推断；
7. `alternatives`：是否有更低成本、更可逆或更安全的路径；
8. `stop_conditions`：出现什么情况必须停止或升级给用户。

不得只从“要做哪些步骤”出发，必须先说明步骤与目标之间的因果链。

## 三、任务发布前影响预测

每个任务必须附带 `TaskImpactForecast`，至少包含：

### 1. 预期正面收益

- 直接收益；
- 间接收益；
- 可复用资产；
- 对其他模块的联动收益；
- 长期复利价值；
- 每项收益的发生概率、影响程度、证据依据和验证方法。

### 2. 预期负面影响

- 技术债；
- 复杂度上升；
- 维护、算力、时间和Token成本；
- 错误架构锁定；
- 数据污染、权限、安全或隐私风险；
- 对既有任务的干扰；
- 误判、过拟合、虚假完成感和叙事幻觉；
- 每项负面影响的概率、严重度、可逆性、可检测性和传播范围。

### 3. 净价值判断

使用定性或半定量方法评估：

```text
expected_net_value
= expected_benefit
- expected_harm
- execution_cost
- opportunity_cost
```

数字不足时不得伪造精确分数，可以使用 `low / medium / high / critical` 并解释依据。

## 四、风险门禁与用户告知

GPT先自行审查预测结果。只有满足以下任一条件时，必须在发布任务前向用户说明并请求决策：

1. 可能造成不可逆数据损失、权限扩大、凭证泄露或真实资金影响；
2. 单项风险严重度为 `critical`；
3. 高严重度且概率不低于 `medium`；
4. 风险可能跨模块传播，影响第二大脑、交易系统或权威数据源；
5. 任务可能改变硬风控、权威蓝图、系统记录源或生产运行方式；
6. 预期负面价值可能大于正面价值；
7. 无可靠回滚或无法确认影响边界。

一般低风险和可逆风险不必频繁打断用户，但必须写入任务记录和验收对比。

## 五、执行期观测合同

执行者必须在Agent执行反馈v2基础上补充：

- 实际出现的正面效果；
- 实际出现的负面效果；
- 新增成本和复杂度；
- 原预测中未出现的意外事件；
- 哪些预测信号已经出现；
- 哪些停止条件接近或触发；
- 为降低损害采取了什么控制；
- 是否发现更优替代路径。

重大意外损害不能等任务完成后才报告。

## 六、交付后预测校准

GPT验收时必须建立 `OutcomeCalibrationReview`，逐项比较：

| 对比对象 | 必须判断 |
|---|---|
| 预期收益 | 实现、部分实现、未实现、无法验证 |
| 预期负面效果 | 未发生、按预期发生、低于预期、高于预期 |
| 意外收益 | 内容、强度、证据和成因 |
| 意外损害 | 内容、强度、证据和成因 |
| 执行成本 | 低于、接近或高于预期 |
| 时间与复杂度 | 预测偏差及原因 |
| 风险控制 | 是否有效、是否产生副作用 |
| 替代方案 | 是否证明更优或更差 |
| 总体净价值 | 正、负、中性或证据不足 |

必须区分：

- `prediction_error`：预测本身错误；
- `execution_error`：执行偏离正确预测；
- `environment_change`：环境变化导致结果不同；
- `measurement_error`：观测或指标错误；
- `unknown_unknown`：任务前无法合理识别的新因素。

## 七、意外正收益强化机制

发现意外正收益时，不得只记录“效果很好”。必须分析：

1. 直接触发因素；
2. 中间因果链；
3. 哪些必要条件存在；
4. 哪些充分条件仍未证明；
5. 是否可能只是偶然、选择偏差或测量误差；
6. 可复用到哪些任务；
7. 放大收益可能带来的新风险；
8. 下一次最小验证实验。

正反馈状态：

```text
OBSERVED_POSITIVE_SURPRISE
→ CAUSAL_HYPOTHESIS
→ REPLICATED
→ STANDARD_CANDIDATE
→ STANDARDIZED
```

单次成功默认只能进入 `CAUSAL_HYPOTHESIS`，不能直接升级为长期标准。

## 八、意外负面结果纠偏机制

实际损害大于预期时，必须进行无责根因复盘：

1. 哪个原始假设错误；
2. 哪个风险信号被遗漏、低估或误读；
3. 任务设计、执行、工具、环境或验收哪个环节失效；
4. 为什么现有门禁没有阻止；
5. 是否存在激励偏差、确认偏差或完成度幻觉；
6. 哪条规则、模板、测试或停止条件需要修改；
7. 下次如何更早检测；
8. 是否需要加入禁止清单或强制用户审批。

负反馈状态：

```text
NEGATIVE_SURPRISE
→ CONTAINMENT
→ ROOT_CAUSE
→ CONTROL_UPDATE
→ REGRESSION_TEST
→ VERIFIED_CORRECTION
```

不得只写“下次注意”。必须产生可执行的系统变化。

## 九、工程经验对象

每条经验保存为 `EngineeringLearningRecord`：

- `learning_id`；
- `source_task`、Issue、PR和提交；
- 预测内容；
- 实际结果；
- 偏差类型；
- 因果假设；
- 支持和反对证据；
- 适用条件；
- 失效条件；
- 风险和副作用；
- 复现次数；
- 当前成熟度；
- 应更新的模板、蓝图、测试、路由或禁令；
- owner、下一验证任务和复查日期。

成熟度：

- `OBSERVATION_ONLY`
- `HYPOTHESIS`
- `PARTIALLY_REPLICATED`
- `REPLICATED`
- `STANDARD_CANDIDATE`
- `STANDARD`
- `DEPRECATED`
- `CONTRADICTED`

## 十、强制回写目标

学习结果不能只留在评论中。根据影响范围回写：

1. 当前Issue和PR；
2. Agent执行反馈v2；
3. 工程学习登记表；
4. 任务发布模板；
5. `AGENTS.md`或Agent路由协议；
6. 对应专项蓝图；
7. 项目集蓝图集成索引；
8. 测试、门禁、禁止清单和回滚手册；
9. 本地受保护权威蓝图的差异包。

## 十一、角色分工

- **GPT**：任务前影响预测、风险审查、交付校准、经验抽象和制度回写；
- **Codex**：技术因果链、代码/架构风险、测试证据和实施经验；
- **WorkBuddy**：真实环境副作用、部署成本、运行态意外和现场验证；
- **QCLAW**：吸收外部研究和内部经验，形成候选规律，不直接升级为权威规则；
- **用户**：高风险、权威蓝图变化、真实资金和重大取舍的最终决定者。

## 十二、成功标准

系统生效后，每个重要任务都能回答：

1. 发布前预期带来什么收益和损害；
2. 为什么认为这些结果会发生；
3. 哪些风险需要用户知道；
4. 实际交付与预测有何偏差；
5. 偏差来自预测、执行、环境还是测量；
6. 哪些意外收益值得复制；
7. 哪些意外损害已经转化为新门禁；
8. 下一任务具体继承了哪些经验。

本系统的最终目标不是减少所有失败，而是让每次成功和失败都能降低未来的不确定性，并持续改善整个工程决策链。
