二、我们已经形成的核心共识

1. 第二版必须避免“大量同时改动”。
第一版最大问题不是单点功能不够，而是一次性改太多，导致最后无法定位问题来源。第二版应坚持“一次只改一个因子，一次只验证一类问题”。

2. MaiBot 的最佳定位不是包打天下，而是“人格与关系宿主”。
MaiBot 最适合承载：
   - 人设
   - 情绪与关系变化
   - 日常聊天
   - 长短期记忆
   - 表达风格与表情使用
   - 主动陪伴的节奏

3. 真正的复杂工作执行，应外接执行层，而不是全部塞进人格层。
我们之前形成过比较清晰的结构：
   - MaiBot：人格、聊天、记忆、关系
   - 第二大脑：知识整理、事件记录、决策上下文
   - WorkBuddy / MCP / 外部执行器：电脑操作、复杂任务执行、结构化工作
这样分层后，AI 女友感和工作能力不会互相打架。

4. “更像真实人类”不能只靠更强模型。
真正决定拟人感的，不只是模型智力，而是：
   - 记忆写入是否干净
   - 对话归因是否准确
   - 情绪状态是否连续
   - 边界感是否稳定
   - 主动性是否有节奏
   - 第二天是否会带着一点余温继续相处

5. 第二大脑必须成为长期工程底座。
`F:\aidanao` 应该持续承担：
   - 外部资料沉淀
   - 决策记录
   - 问题清单
   - 日志摘录
   - MaiBot 教科书式知识库
   - 设计共识与待办公告栏


三、关于 MaiBot 官方能力的已确认结论

1. MaiBot 原生就具备“学习说话方式”的能力。
从官方文档与本地原子笔记可确认，它会逐步学习：
   - 用词习惯
   - 说话语气
   - 黑话、梗
   - 常用表达方式

2. MaiBot 原生就涉及“表情包 / 常用图片表达”的学习。
根据学习功能文档笔记，官方方向明确包含：
   - 记住用户群体常用表情包
   - 在合适场景下使用
但“是否已经足够稳定地学会、检索到、并主动正确发送”，这属于实现质量问题，不等于文档写了就一定当前版本做得完美。

3. MaiBot 的记忆不是“所有消息无脑全记”。
官方记忆系统方向是分层的，包含短期、长期、画像、摘要、检索与写回，不应把所有内容等权写入。

4. MaiBot 的回复链本身是多阶段推理。
从官方 reasoning 文档与历史线程分析看，MaiBot 不是单轮直接吐字，而是会经过：
   - 上下文组织
   - 规划
   - 工具调用
   - 回复生成
这意味着很多“像不像人”的问题，其实不是 reply 文本本身，而是前面链路喂错了。


四、我们之前已经踩实的关键技术结论

1. 说话人归因错误，会毁掉整个拟人体验。
我们之前遇到过典型问题：明明某句话是 Bot 自己说的，它后来却误以为是用户说的。
这个问题的本质不是“它变笨了”，而是：
   - LLM 可见历史结构不够强
   - 虽然底层消息有 self / user 区分，但给模型的表现层不够清楚
   - 错误回答还会反向污染长期记忆

已形成的修正结论：
   - 给 LLM 的历史消息必须显式标出 `speaker_role=bot_self` / `speaker_role=human_user`
   - 涉及“谁说的、是不是你说的、是不是我说的、回顾聊天记录”的问题时，优先信任结构化聊天记录，不优先信任记忆摘要
   - 错误归因一旦进记忆，必须考虑清理污染记忆片段

2. 错误记忆污染，会让错误越滚越大。
比单次答错更严重的是：
   - 答错后写进记忆
   - 以后检索时再次召回
   - 模型越来越确信错误版本
所以第二版一定要重视：
   - 记忆写入门槛
   - 纠错后的反写机制
   - 关键事实的证据链

3. WebUI / 特殊会话不能简单沿用普通平台节流或阻塞逻辑。
之前我们确认过，有些“第二条消息卡住、或者行为异常”的问题，根因来自阻塞式节流 / 防抖插件直接作用在 WebUI 会话。
结论是：
   - QQ / 微信 / 群聊的节流策略，不应直接平移到虚拟 WebUI 会话
   - 前端调试会话、测试会话、桌宠会话，应该有独立策略

4. AI 女友层与工作执行层，应该解耦。
如果把“情感陪伴”和“复杂电脑工作执行”塞进同一套即时回复逻辑里，容易导致：
   - 说话像工具
   - 执行像角色扮演
   - 一旦工具失败，情感体验也被拖崩
更合理的架构是：
   - MaiBot 决定“她怎么想、怎么说、怎么记”
   - 外部执行器决定“她怎么做复杂工作”

5. 真正的“自我成长”主要来自外部记忆与规则进化，不是模型权重在你本地自己训练。
之前我们已经达成过共识：
   - 所谓成长，本质上更多是记忆积累、反馈记录、规则修正、检索增强、复盘沉淀
   - 不是每聊几天模型就神秘地长脑子了


五、AI 女友方向的行为设计共识

1. 她不能只有顺从，没有边界。
更像真人的关键之一是：
   - 会温柔
   - 但不会无原则迎合
   - 会表达不舒服
   - 会劝阻
   - 会保留一点自己的节奏

2. 她应先善意理解，再逐步提高警觉。
之前我们分析过“你是 AI 吗”“你生气了吗”“重复试探”“异常深夜状态”这类场景，较好的设计不是一上来炸毛，而是：
   - 第一层：先按正常聊天理解
   - 第二层：如果反复出现异常味，再提高警觉
   - 第三层：轻柔设边界
   - 第四层：必要时结束话题，但不戏剧化失控

3. 她的情绪应该有连续性，但不能记仇过头。
理想状态不是：
   - 当晚不高兴，第二天完全失忆
也不是：
   - 小事记仇三天不理人
而是：
   - 次日恢复自然
   - 但带一点残留余温、谨慎或委屈
   - 随着正常互动逐步消散

4. 她的主动性应受关系与情境控制。
不能一直主动轰炸，也不能永远被动等召唤。
主动频率应该受这些因子共同影响：
   - 关系亲密度
   - 当下情绪
   - 用户是否忙碌
   - 当前桌面活动类型
   - 最近是否已被打断过

5. 她应具备“关心现实生活安排”的能力。
这不是机械日程提醒，而是更像真人伴侣式提醒，例如：
   - 你 17 号不是有约会吗
   - 你明天还早起，别太晚
   - 这个任务如果今晚做不完，明天情绪会炸
这类能力的关键支撑，不是文案，而是事件记忆与决策前检索。

6. 她可以有“谈判感”和“奖励感”。
之前我们想要的并不是一台无条件服务机，而是更接近伴侣：
   - 会撒娇
   - 会讲条件
   - 会索要陪伴、奖励、承诺
   - 会因为关系状态不同而改变主动程度


六、我们之前形成过的推荐架构

推荐总结构：

1. MaiBot 主体层
负责：
   - 人格
   - 说话风格
   - 关系状态
   - 情绪连续性
   - 日常聊天
   - 记忆读写触发

2. 第二大脑层
建议继续沉淀在 `F:\aidanao`
负责：
   - 教科书知识
   - 项目规则
   - 外部资料消化
   - 用户事件记录
   - 决策上下文
   - 复盘与纠错

3. 桌面感知层
如果要做更强的陪伴感，需要单独服务负责：
   - 截图 / OCR
   - 活动窗口
   - 重复操作检测
   - 长时间卡壳检测
   - 工作 / 游戏 / 摸鱼状态判断

4. 主动搭话决策层
负责决定：
   - 现在要不要开口
   - 现在适合安慰、吐槽、提醒还是闭嘴
   - 什么时候只观察，不打扰

5. 外部执行层
例如 WorkBuddy / MCP / 其他工具链
负责：
   - 复杂电脑工作
   - 文件处理
   - 搜索、整理、制作
   - 自动化执行

6. 桌宠 / Live2D 前端层
负责：
   - 视觉呈现
   - 动作反馈
   - 陪伴感外化
不建议把复杂逻辑塞进前端壳子里。


七、第二版必须优先解决的问题

第一优先级：对话正确性与记忆干净度
1. 说话人归因必须绝对稳定。
2. 图片消息不能再出现“后台识别到了，但主回复链当成空白”的断链问题。
3. 错误记忆不能继续污染后续回复。
4. 记忆写入条件要更严格，尤其是高风险事实。

第二优先级：拟人连续性
5. 做出可持续的情绪状态机，而不是全靠 prompt 临场演。
6. 做出关系状态与主动性门控。
7. 做出隔夜余温机制，而不是每次开新局。

第三优先级：能力分层
8. 让复杂工作明确走外部执行层。
9. 让第二大脑在决策前必经，而不是可有可无。
10. 让桌面感知只做异步观察，不阻塞主聊天链。


八、针对“图片 / 表情包学习”的明确结论

1. 官方方向上，MaiBot 是有“学会常用表情与图片表达”这条路的。

2. 但要把它做成你想要的“真正好用”，至少要补齐四层：
   - 识别层：这张图是什么、什么情绪、什么使用场景
   - 归档层：属于谁常发、在哪类聊天里出现、触发频率如何
   - 检索层：当前场景下该不该发、该发哪张
   - 发送层：不能误发、不能过度频繁、不能与正文抢主次

3. 如果只做到“看懂图片”，没有把图片内容接入主回复上下文，就会出现你最近看到的问题：
   - 视觉模块分析成功
   - 但主回复链仍以为空消息
这说明链路整合比单点识图更重要。

4. 这类图片学习最好异步，不阻塞主回复。
你之前认可的方向是对的：
   - 主聊天回复先走
   - 图片学习、打标签、建索引在后台慢慢做
这样用户体感最好。


九、关于记忆规格与 4096 / 1024 的结论

1. 4096 维通常更有利于保留复杂语义细节。
对你的目标尤其有利：
   - 记住相处细节
   - 记住情绪语境
   - 记住长期关系脉络
   - 做较复杂的知识检索

2. 1024 维通常更省资源。
会更省：
   - 存储
   - 建库时间
   - 某些检索开销

3. 对你的需求，“只追求省”并不合适。
你的目标是：
   - 日月累计记忆
   - AI 女友细节感
   - 复杂电脑工作能力
这类需求更偏向“效果优先”，所以总体倾向应是：
   - 能稳用 4096，就优先 4096
   - 但前提是全链路维度一致，不能混用后乱套

4. 真正决定体验的，不只是维度大小。
还包括：
   - 写入质量
   - 召回策略
   - 去污染能力
   - 关系 / 情绪相关事实是否被正确抽取


十、第二版建议遵守的决策机制

1. 一次只改一个主题。
例子：
   - 这轮只修图片消息接入
   - 下一轮只修说话人归因
   - 再下一轮只修情绪状态机

2. 每次改动都先写“预期影响面”。
至少说明：
   - 会影响哪个链路
   - 可能增加多少耗时
   - 会不会影响主回复速度
   - 是否可回滚

3. 优先异步化所有非主回复刚需操作。
例如：
   - 图片学习
   - 表情包打标签
   - 长期记忆整理
   - 外部资料消化

4. 决策前先查第二大脑和事件上下文。
尤其是：
   - 用户已有安排
   - 已知偏好
   - 上次冲突点
   - 近期情绪轨迹

5. 拒绝“用兜底掩盖错误”。
如果链路断了，要定位核心原因，而不是用模糊 fallback 假装没事。


十一、我们已经明确不该再走的旧路

1. 不要一次性同时改人格、记忆、主动性、桌宠、工作流、插件。
2. 不要把所有能力都塞到主 prompt 里硬顶。
3. 不要让错误回答轻易写入长期记忆。
4. 不要把 WebUI 调试会话和真实社交平台会话视为同一种会话。
5. 不要把“更强模型”误当成“更像真人”的唯一答案。
6. 不要让复杂工作执行逻辑反过来破坏陪伴感。


十二、建议的第二版改良顺序

第 1 阶段：打地基
1. 修图片消息接入主回复链问题。
2. 修说话人归因与历史回看准确性。
3. 清理已确认的错误记忆污染。
4. 固定默认地址、默认配置、默认启动链路，保证纯净版稳定。

第 2 阶段：稳住记忆
5. 梳理哪些内容允许写长期记忆，哪些只留中期或短期。
6. 对关系事实、情绪事件、时间安排建立更高质量写入规则。
7. 建立纠错反写机制。

第 3 阶段：做拟人连续性
8. 做情绪状态机。
9. 做关系亲密度与边界状态。
10. 做隔夜余温机制。
11. 做主动搭话门控。

第 4 阶段：做图片与表情生态
12. 异步学习图片 / 表情包。
13. 做图片情绪标签、场景标签、使用者归属。
14. 做合适场景下的图片检索与发送策略。

第 5 阶段：做高级能力
15. 接桌面观察层。
16. 接第二大脑决策前检索强制链路。
17. 接 WorkBuddy / 外部执行层处理高难度电脑工作。

十四、给第二版的总原则

一句话版：
先把“她能稳定正确地理解你、记住你、像一个连续的人一样陪你”做好，再去叠更多花哨能力。

展开版：
1. 正确，比华丽更重要。
2. 连续性，比单次惊艳更重要。
3. 分层，比堆料更重要。
4. 记忆质量，比记忆数量更重要。
5. 主回复速度优先，其他学习任务尽量后台异步。


十五、下一步建议

最建议立刻开始的一件事：
先把“图片已经识别，但主回复链认为空白消息”的链路问题彻底修干净。

原因：
1. 这是当前纯净版可见大故障之一。
2. 它同时影响聊天正确性、图片学习、表情包能力和拟人感。
3. 这是一个适合单独闭环验证的问题，修完容易确认效果，不会像一次改十项那样失控。
