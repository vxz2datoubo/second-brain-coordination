# L4 进化引擎 · 纠正→技能转化流水线

## 理论基础

### 1. Donald Schön · 反思性实践 (Reflective Practice)

| 反思类型 | 在本系统中的体现 |
|---|---|
| **行动中反思** (reflection-in-action) | AI 生成 ShotManifest 时，如果自己发现某个表演方向与上下文矛盾 → 在生成过程中修正，不等到用户指出 |
| **行动后反思** (reflection-on-action) | 用户纠正后 → 对比初始版→修改版→最终版的演变 → 追问"为什么第一版是错的" |
| **关键约束** | 不能只保存最终成品。每次纠正必须保存：原输出、用户纠正的原文、AI 当时的错误假设、修改判断过程 |

### 2. Chris Argyris · 双环学习 (Double-loop Learning)

| 学习类型 | 在本系统中的体现 |
|---|---|
| **单环学习** (改表面) | "奥兹不应该轻蔑抱怨 → 把这句删掉" → 只改了这一镜，下次遇到类似情况继续犯错 |
| **双环学习** (改假设) | "为什么 AI 会把社交措辞当成情感流露？→ 因为默认假设是'台词=角色真实想法' → 这个假设在权力不对等场景下是错的 → 修正假设：权力不对等时，台词优先解读为社交工具" |
| **关键约束** | 纠正不能只改一句台词——必须追溯到**产生错误的那条默认假设**，修正它。 |

### 3. 设计型研究 (Design-Based Research)

循环：**分析 → 设计 → 实施 → 评估 → 修订 → 提炼可迁移设计原则**

| 阶段 | 在本系统中的体现 |
|---|---|
| 分析 | 用户纠正了什么？为什么原方案不行？ |
| 设计 | 怎么改？新方案基于什么原理？ |
| 实施 | 更新 ShotManifest + 重新生成 prompt |
| 评估 | 用户是否接受新方案？还是继续纠正？ |
| 修订 | 根据评估微调技能规则 |
| 提炼 | 泛化为可迁移原则 → 写入 YAML 技能文件 |

---

## 定位
L4 进化引擎是 film-pipeline 的**元进程**——不是技能，是产技能的机器。
每次用户纠正 AI 的推论或原理，进化引擎自动跑一次完整闭环。

## 完整流程

### 第 1 步：记录纠正

```
输入：用户说"不对，这里应该是……"
动作：
  ① 记录原始输出（AI 原来怎么写的）
  ② 记录用户纠正的具体内容
  ③ 记录上下文（哪个场景、哪个角色、什么情绪）
  ④ 判断纠正类型：表演方向 / 镜头选择 / 灯光方案 / 叙事改编 / 平台合规
输出：纠正卡片
```

### 第 2 步：反推原理

```
输入：纠正卡片
动作：
  ① 问"AI 为什么会犯这个错？"——找到错误假设
  ② 问"用户为什么这么改？"——反推用户的导演直觉
  ③ 抽象为可迁移原则——"这个纠正只适用于奥兹，还是适用于所有权力不对等的对话？"
  ④ 写成一句话原则（能放进 YAML 的 principle 字段）
输出：原则草案
```

### 第 3 步：检索学术依据

```
输入：原则草案
动作：
  ① 提取关键词 → WebSearch 搜索导演学/表演学/电影理论
  ② 找到匹配的学术概念（Stanislavski / Subtext / Mise-en-scène / etc.）
  ③ 验证：学术概念是否支持用户纠正的方向？
  ④ 如果找到学术支撑 → 录入 academic_basis
  ⑤ 如果学术不支撑 → 标记为"用户经验规则"，降级为 draft 状态
输出：经过学术验证（或经验标记）的原则
```

### 第 4 步：生成原子技能

```
输入：经过验证的原则
动作：
  ① 定义 triggers（什么场景触发这个技能）
  ② 写 diagnosis 诊断公式（怎么判断需要这个技能）
  ③ 写 apply 应用步骤（怎么做）
  ④ 写 before/after 案例
  ⑤ 存为 skills/{skill-id}.yaml（YAML 格式，机器可读）
输出：原子技能文件
```

### 第 5 步：写入索引

```
输入：原子技能文件
动作：
  ① 在 SKILL-INDEX.yaml 的 trigger_map 中添加触发映射
  ② 在 skill_registry 中注册技能（含 correct_count）
  ③ 如果已有同类技能 → 合并或升级版本
  ④ 确保 trigger_map 的 condition 足够精确（不过度触发）
输出：更新后的 SKILL-INDEX.yaml
```

### 第 6 步：运行时检索（管线生成时自动执行）

```
输入：当前场景的文本、角色关系、情绪标签
动作：
  ① 扫描触发条件：
     - 是否有权力不对等关系？ → 查 trigger_map.power_gap_dialogue
     - 是否涉及暴力？ → 查 trigger_map.violence
     - 角色是否在说谎/隐藏情绪？ → 查 trigger_map.subtext
  ② 加载匹配的技能 → Read skills/{skill-id}.yaml
  ③ 在生成该场景的 ShotManifest 时，静默注入技能规则
  ④ 如果技能有 counter_example → 加入该模块的 negative_prompt
输出：增强了技能约束的 ShotManifest
```

### 第 7 步：闭环跟踪

```
每次应用技能后：
  ① 技能文件的 last_applied 更新为当前日期
  ② correct_count +1（首次纠正=1，后续每次成功应用+1）
  ③ 如果应用后用户又纠正 → 检查是技能规则不够精确，还是场景不匹配 → 更新技能或新增
  ④ correct_count >= 3 且连续无纠正 → 状态升为 已锁定
```

## 与现有架构的关系

```
用户纠正
  ↓
[L4 进化引擎] ← 本文件
  ├─→ 原则 → skills/{skill-id}.yaml → SKILL-INDEX.yaml
  └─→ 修正 → modules/XX-*.md 进化日志

管线运行时:
  SKILL-INDEX.yaml (扫描触发条件)
    ↓ 匹配的技能
  skills/{skill-id}.yaml (加载规则)
    ↓ 注入
  [01→09 模块] (生成 ShotManifest 时应用技能约束)
```

## 状态机

```
用户纠正 → [draft: 刚记录，未验证学术] 
  → [验证通过] → [待确认: 有学术支撑] 
  → [使用 3 次无纠正] → [已锁定: 可靠]
或
  → [验证未通过] → [用户经验: 无学术支撑但用户确认有效]
  → [使用 3 次无纠正] → [已锁定]
或
  → [后续使用中出现纠正] → [待修正] → [更新版本]
```

## 技能的生命周期

```
draft → 待确认 → 已锁定 → (废弃)
         ↑__________|  (被反例推翻后存档，保留 30 天)
```

## 与 AI 对话的接口

进化引擎在 AI 对话中的表现：

> 用户："不对，奥兹不应该……"
> AI：
>   "收到。（第1步：记录纠正）你纠正的是：奥兹的台词应解读为社交恭维而非情感流露。
>   （第2步：反推原理）原理是：权力不对等时，台词=社交工具。
>   （第3步：检索学术）对应 Stanislavski 的 Objective vs Tactic + Subtext 概念。
>   （第4步：生成技能）已写入 skills/dialogue-as-social-tool.yaml。
>   （第5步：更新索引）已注册到 SKILL-INDEX。
>   以后遇到权力不对等的对话场景，我会自动调取这个技能。"
>   → 同时修正当前 ShotManifest

**关键原则：修正当前产出 + 生成未来可复用技能，两步同时做。不等到"以后"。**
