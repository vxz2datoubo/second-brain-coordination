# AI任务完成、异常与新发现反馈协议 v1.0

适用对象：Codex、WorkBuddy  
项目：F:\aidanao  
目标：减少无意义转发，同时确保失败、风险、新发现和改进建议不会丢失。

---

## 1. 核心原则

任务结束后必须生成一张“结果收据”，但不必每次都让用户转发给ChatGPT。

规则：

```text
正确完成 + 验收全部通过 + 无新增问题
→ 简短收据，写入本地结果目录
→ 不需要转发ChatGPT

正确完成，但发现新问题/新机会/规则冲突
→ 必须报告
→ 建议用户转发ChatGPT

部分完成、阻塞、失败、需要审批
→ 必须报告
→ 必须转发ChatGPT或用户处理
```

任务是否完成，与是否有新发现，是两个独立维度。

即使任务成功，也可能同时发现值得上报的问题。

---

## 2. 统一状态

只允许使用以下状态：

### SUCCESS_CLEAN

全部验收通过，无新增风险、冲突或重要建议。

处理：

- 返回简短收据；
- 保存本地；
- 用户无需转发ChatGPT。

### SUCCESS_WITH_FINDINGS

任务正确完成，但发现了新的风险、机会、架构问题、数据问题或改进建议。

处理：

- 返回完整反馈；
- 用户应转发ChatGPT；
- 重要内容进入INBOX或记忆补丁候选。

### PARTIAL

只完成部分目标，剩余部分不影响已完成产物，但仍需后续任务。

处理：

- 说明已完成与未完成；
- 提供阻塞原因和下一步；
- 用户应转发ChatGPT。

### BLOCKED

由于权限、数据、依赖、规则冲突或缺少批准，无法继续。

处理：

- 不得猜测或绕过；
- 报告需要谁提供什么；
- 用户必须转发。

### FAILED

执行失败，产物不可用或验收未通过。

处理：

- 报告错误、影响、回滚状态和重试建议；
- 用户必须转发。

### NEEDS_APPROVAL

已经准备完成，但下一步涉及实盘、密钥、删除、覆盖、生产部署或其他高风险操作。

处理：

- 停止执行；
- 只列出待批准动作；
- 用户亲自决定。

---

## 3. 新发现分类

任务完成时必须检查是否发现以下内容：

- DATA_QUALITY：缺失、乱序、字段口径、时间戳或数据源冲突
- ARCHITECTURE：模块边界、重复系统、技术债或扩展问题
- RESEARCH：新假设、证据反例、回测缺口
- MARKET_RULE：A股规则变化、历史规则版本问题
- SECURITY：密钥、权限、数据泄露或危险操作
- LIVE_RISK：可能影响实盘、仓位、订单或风控
- PERFORMANCE：延迟、吞吐、存储和资源问题
- DOCUMENTATION：正式规则和实际代码不一致
- OPPORTUNITY：值得新增的接口、技能、模型或研究方向
- MEMORY_PATCH：值得进入永久记忆的结论

---

## 4. 严重度

### P0

涉及：

- 真实资金
- 密钥或账户安全
- 错单或可能自动下单
- 数据泄漏
- 回测存在严重未来函数
- 市场规则错误可能导致交易错误

必须立即停止并上报用户和ChatGPT。

### P1

阻塞当前主任务，或可能导致数据、模型、架构结论错误。

必须上报ChatGPT。

### P2

不阻塞当前任务，但值得在近期修正或研究。

加入任务队列，建议批量转发ChatGPT。

### P3

小优化、文档建议、低优先级机会。

本地记录即可，可在周期总结中汇总。

## 5. SUCCESS_CLEAN简短收据

任务正确完成时，只需返回：

```text
【任务ID】
【状态】SUCCESS_CLEAN
【验收】全部通过
【主要产物】
【测试】通过项/总项
【新增问题】无
【结果路径】
【SHA256，如适用】
```

用户看到这类回复后，不需要再转发ChatGPT。

## 7. 本地保存

WorkBuddy负责保存：

```text
F:\aidanao\coordination\RESULTS\<TASK_ID>\RESULT-RECEIPT.md
F:\aidanao\coordination\RESULTS\<TASK_ID>\RESULT-RECEIPT.json
```

若有新发现，同时写入：

```text
F:\aidanao\memory\INBOX.md
```

仅当ChatGPT确认值得长期保存后，才生成Memory Patch。

Codex在聊天中返回同样格式；若已经能写入仓库，则同时保存到任务结果目录。

---

## 8. 不需要重复报告的内容

以下情况不必展开：

- 全部验收通过；
- 没有代码或数据冲突；
- 没有新风险；
- 没有新建议；
- 产物路径和测试结果清晰；
- 不涉及用户审批。

不要为了显得认真重复任务说明或写长篇总结。

---

## 9. 必须主动报告的“顺带发现”

即使不属于当前任务，只要满足以下任一条件，也必须报告：

- 可能导致未来错误；
- 会影响其他模块；
- 与正式文档冲突；
- 有明显复用价值；
- 可以显著节约成本或时间；
- 值得加入永久记忆；
- 需要用户改变方向或批准。

但不得在当前任务中擅自扩大范围修改。

正确做法：

```text
发现
→ 记录
→ 给出证据和建议
→ 等待新任务
```

---

## 10. 谁需要回复谁

### Codex或WorkBuddy → 用户

每个任务都必须有结果收据。

### 用户 → ChatGPT

以下才需要转发：

- 非SUCCESS_CLEAN；
- SUCCESS_WITH_FINDINGS；
- 任何P0/P1；
- 需要新架构决策；
- 需要生成记忆补丁；
- 用户自己不确定是否完成正确。

### ChatGPT → Codex/WorkBuddy

只有需要：

- 修订；
- 新任务；
- 验收；
- 规则更新；
- 架构调整；

才再次下发文件或指令。

---

## 11. 目标效果

让反馈系统像交通灯：

- 绿灯：已完成，无需转发；
- 黄灯：完成但有发现，转发审核；
- 红灯：阻塞、失败或高风险，立即停止并上报。

减少聊天噪声，但不让关键问题掉进地板缝里。
