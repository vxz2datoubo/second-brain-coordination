# 注意力信号引擎设计蓝图

> 文档状态：Proposed v0.1  
> 所属项目集：企业级 A 股 AI 量化交易与主导行为机制推断平台 / 超级第二大脑  
> 模块定位：市场关注度与信息扩散分析侧车模块  
> 默认交易状态：`NO_TRADE`  
> 单一真相源：本 GitHub 仓库中的受控版本  
> 最后更新：2026-07-18

## 1. 文档目的

本文件把“主动搜索通常比内容全面扩散更早暴露真实意图”的观察，固化为可采集、可验证、可回测、可扩展的工程能力。

模块暂定名为 **注意力信号引擎（Attention Signal Engine，ASE）**。它不是一个简单的热搜聚合器，也不是看到某个话题变热就发出买卖指令的黑箱。它要回答的是：

1. 哪些技术、产业、产品或公司主题正在被人主动寻找。
2. 主动搜索是否领先于内容生产、被动消费、媒体报道和资本市场反应。
3. 热度来自少量专业人群，还是已经进入大众扩散。
4. 热度是有效信息、营销投放、情绪冲击，还是刷量和短期噪声。
5. 某个主题如何映射到 A 股公司、产业链、公告、研报和价格行为。
6. 这种传播路径在历史上是否具有稳定、可重复的研究价值。

## 2. 核心假设

### 2.1 待验证的主假设

对于“世界模型”这类早期技术主题，传播可能经历以下链路：

专业圈层出现信号 → 少量高意图用户主动搜索 → 内容创作者增加供给 → 平台推荐扩大被动触达 → 媒体与券商跟进 → 资本市场形成概念映射 → 交易拥挤或叙事衰减。

这个顺序不能被写死。系统必须通过历史数据检验各平台之间的领先和滞后关系。不同主题、不同平台、不同市场阶段可能出现相反顺序。

### 2.2 信号分层

注意力被拆为六类，不使用一个混合热度值掩盖差异：

- **主动需求**：用户主动输入关键词、查找解释、比较产品或寻找投资映射。
- **内容供给**：新增视频、笔记、文章、问答、直播和账号数量。
- **传播消费**：播放、阅读、完播、停留、收藏、评论、转发和互动。
- **专业确认**：论文、代码仓库、模型下载、专利、招聘、开发者讨论和产业会议。
- **媒体与机构确认**：新闻报道、公众号文章、券商研报、公司公告和调研纪要。
- **资本市场同步**：相关股票成交、价格、波动、板块联动、资金与研报覆盖变化。

## 3. 系统边界与治理原则

### 3.1 侧车定位

ASE 属于研究与特征侧车，不进入真实订单关键路径。它可以输出标准化特征、证据链和研究报告，但不能绕过确定性策略、组合风控、数据质量门禁和人工授权直接下单。

### 3.2 默认安全状态

在完成数据质量验证、历史回测、稳定性测试和独立验收前，所有输出必须标记：

```yaml
trade_action: NO_TRADE
research_only: true
```

### 3.3 不编造数据

平台未提供公开接口、账号无权限、页面不可稳定访问或口径不透明时，必须记录为未知或不可用。严禁用搜索结果页的模糊数字冒充官方指数，严禁把第三方估算伪装成平台原始数据。

### 3.4 合规与最小采集

采集器必须遵守平台条款、访问频率限制、版权要求和个人信息保护原则。优先使用官方页面、官方导出、授权 API 和人工录入。未经评估不得部署绕过登录、验证码或反自动化机制的抓取方案。

## 4. 与现有项目集的关系

本模块遵守“一个项目集、两个独立产品、一个受控联动层”：

- 交易系统独立运行。
- 第二大脑只读接入研究工件、证据链和主题演化结果。
- ASE 输出进入研究特征层，不进入订单热路径。
- Codex 负责核心架构、数据契约、特征引擎、测试和重构。
- WorkBuddy 负责真实数据接入、账号权限验证、定时运行、监控和故障回传。
- ChatGPT 负责研究设计、来源审计、任务拆解、独立验收和跨模块协调。
- 用户负责资金、高风险操作、付费数据源、账号授权和最终策略批准。

## 5. 总体架构

```text
主题注册表
   ↓
平台适配器与人工录入
   ↓
原始事实层 Raw Facts
   ↓
口径归一化与数据质量门禁
   ↓
注意力特征工厂
   ↓
扩散阶段识别与领先滞后分析
   ↓
A股产业链与公司映射
   ↓
研究报告 / 看板 / 回测特征 / 第二大脑只读工件
```

### 5.1 主题注册表

每个研究主题必须有稳定 ID、关键词词包、别名、歧义词、排除词、关联技术、关联产品和 A 股映射候选。

示例：

```yaml
topic_id: tech.world_model
canonical_name: 世界模型
aliases:
  - world model
  - 物理世界模型
  - 生成式世界模型
  - 空间智能
related_terms:
  - 具身智能
  - 视频生成
  - 机器人基础模型
negative_terms: []
status: CANDIDATE
```

所有词包调整必须保留版本，以避免回测时发生“今天的词表解释昨天的数据”。

### 5.2 平台适配器

每个平台独立适配，不把不同口径强行混为一谈。适配器至少输出：

- 平台与数据产品名称
- 指标名称和官方定义
- 时间粒度
- 地域范围
- 原始值或指数值
- 采集方式
- 权限状态
- 抓取或录入时间
- 来源链接或截图工件引用
- 数据质量等级
- 失败原因

### 5.3 原始事实层

原始数据只追加，不原地覆盖。修订通过新版本和修订原因表达。任何特征都必须能追溯到原始观察记录。

### 5.4 特征工厂

特征工厂将不同平台的量纲转换为可比较的变化信号，例如滚动分位数、同比、环比、加速度、异常分、持续性和跨平台一致性。

### 5.5 扩散阶段识别

系统不只输出“热”或“不热”，而是识别主题所处阶段：

| 阶段 | 名称 | 典型表现 |
|---|---|---|
| S0 | 潜伏 | 专业源出现零散信号，大众平台无明显变化 |
| S1 | 专业萌芽 | GitHub、论文、模型下载、招聘或开发者讨论增加 |
| S2 | 主动探索 | 百度、微信、抖音、小红书等主动搜索开始持续上升 |
| S3 | 内容供给 | 新增视频、文章、笔记、问答和账号显著增加 |
| S4 | 大众扩散 | 播放、阅读、互动和跨平台传播加速 |
| S5 | 资本确认 | 研报、公告、投资者互动和相关股票联动增强 |
| S6 | 拥挤或衰减 | 搜索增速转弱，内容仍高位，市场交易拥挤或叙事退潮 |

阶段识别必须输出置信度、证据和反例，不允许只给一个标签。

## 6. 数据源地图

下表描述首批候选源。具体权限和自动化可行性必须由 WorkBuddy 在真实环境验证。

### 6.1 主动搜索与平台指数

| 来源 | 核心价值 | 首选指标 | 初始接入方式 | 备注 |
|---|---|---|---|---|
| 百度指数 | 直接反映百度主动搜索需求 | 搜索指数、移动搜索指数、资讯指数、媒体指数、需求图谱 | 官方页面、人工导出优先 | 官方帮助明确区分搜索、资讯和媒体口径 |
| 微信指数 | 微信生态内主动搜索与综合关注 | 关键词指数、日变化、峰值日期 | 微信内人工记录或授权能力 | 作为高优先级源，接口与导出能力待验证 |
| 抖音指数 | 抖音生态的搜索、内容与传播信号 | 搜索分、内容分、传播分、关联词 | 官方产品页面或小程序人工导出 | 原巨量算数在 2026 年升级为抖音指数，旧功能可用性需重新验收 |
| 小红书聚光及相关官方工具 | 消费决策和种草场景中的搜索意图 | 搜索指数、搜索词、相关词、内容趋势 | 商业账号官方页面或授权 API | 普通账号和商业账号权限可能不同 |
| 微指数 | 微博话题综合热度 | 提及、阅读、互动加权指数 | 移动版人工记录或授权服务 | 网页版状态可能变化，需以实际账号验证 |
| B站 Z 指数及站内搜索 | 年轻技术用户与长视频解释内容 | 内容指数、搜索行为、相关内容量 | 官方指数页面、站内公开数据 | 需要分离内容量、观看和搜索成分 |
| 知乎热榜与站内主题数据 | 高认知成本主题的解释与讨论需求 | 搜索热度、消费热度、互动热度、回答增量 | 官方公开页面或授权数据 | 热榜不等于纯搜索指数，必须保留口径 |
| Google Trends | 海外和全球技术扩散对照 | 相对搜索兴趣、地域、相关查询 | 官方页面或合规接口 | 不代表中国大陆总体搜索需求，仅作外部对照 |

### 6.2 内容供给与传播消费

首批内容平台包括抖音、小红书、B站、微博、微信公众号、知乎、头条、视频号。对每个平台至少记录：

- 新增相关内容数
- 活跃创作者数
- 原创内容比例
- 总播放或阅读
- 中位数播放或阅读
- 互动率
- 收藏率
- 转发率
- 评论语义和问题类型
- 头部内容集中度
- 内容生命周期
- 账号类型分布
- 商业投放或营销痕迹

总量和增量必须同时保存。只看累计内容量会把历史存量误判为当期趋势。

### 6.3 专业与产业确认

| 类别 | 候选来源 | 关键指标 |
|---|---|---|
| 开源 | GitHub | Star 增量、Fork 增量、贡献者、提交频率、Issue 质量、仓库新增数 |
| 模型生态 | Hugging Face 等 | 下载量、收藏、模型新增、衍生模型、更新频率 |
| 学术 | arXiv、会议、论文数据库 | 论文新增、作者机构、引用、主题聚类、会议等级 |
| 专利 | 官方专利数据库 | 申请量、申请人、技术分类、时间滞后 |
| 招聘 | 合规招聘数据源 | 岗位新增、技能词、薪资、城市、公司分布 |
| 产业 | 公司官网、产品发布、开发者大会 | 产品发布、版本、客户案例、合作伙伴 |
| 机构 | 券商研报、行业报告 | 报告数量、首次覆盖、标题词频、观点分歧 |

### 6.4 资本市场映射

对每个主题建立“主题 → 技术环节 → 产业链位置 → 公司 → 证券”的可版本化映射。映射必须区分：

- 核心产品收入相关
- 技术储备或研发相关
- 供应链间接相关
- 市场叙事相关
- 仅名称或概念蹭热点

资本市场特征包括：

- 相关证券收益、成交额、换手率、波动率
- 板块内联动与扩散宽度
- 涨停和异动数量
- 公司公告、互动易、调研和研报变化
- 北向、融资融券、龙虎榜等合规可得资金特征
- 主题热度与价格行为的领先滞后关系

## 7. 数据能力状态机

每一个数据源和指标必须登记能力状态：

| 状态 | 含义 |
|---|---|
| `VERIFIED_OFFICIAL_API` | 官方 API 已验证，口径和权限明确 |
| `VERIFIED_OFFICIAL_EXPORT` | 官方页面可稳定导出或下载 |
| `VERIFIED_OFFICIAL_UI` | 只能通过官方界面人工或半自动采集 |
| `PARTNER_REQUIRED` | 需要商业账号、合作方或付费授权 |
| `EXPERIMENTAL_PUBLIC` | 公开页面可观测，但稳定性和条款待评估 |
| `MANUAL_ONLY` | 暂时只允许人工录入和证据存档 |
| `BLOCKED` | 因权限、合规、验证码或稳定性禁止自动接入 |
| `UNKNOWN` | 尚未完成验证 |

任何 `UNKNOWN` 指标不得进入线上评分。

## 8. 统一数据模型

### 8.1 核心实体

- `TopicDefinition`：主题及词包版本。
- `SourceDefinition`：平台、产品、指标口径和权限。
- `RawObservation`：某时点的原始观察值。
- `ContentObservation`：内容级或聚合级传播事实。
- `AttentionFeature`：归一化后的研究特征。
- `DiffusionStageAssessment`：扩散阶段、置信度和证据。
- `MarketMapping`：主题到公司和证券的映射。
- `DataQualityReport`：缺失、延迟、异常和口径漂移。
- `ResearchArtifact`：日报、周报、图表和事件研究结果。

### 8.2 `RawObservation` 示例

```json
{
  "observation_id": "obs_20260718_baidu_world_model_search",
  "topic_id": "tech.world_model",
  "keyword_version": "v1",
  "source_id": "baidu_index",
  "metric_id": "search_index_mobile",
  "observed_date": "2026-07-18",
  "value": null,
  "unit": "index",
  "geography": "CN",
  "collection_method": "MANUAL_ONLY",
  "capability_status": "VERIFIED_OFFICIAL_UI",
  "source_artifact": null,
  "quality_grade": "PENDING",
  "notes": "示例结构，不代表已获得真实数值"
}
```

### 8.3 存储分层

```text
data/attention/raw/          原始不可变观察
data/attention/normalized/   统一口径数据
data/attention/features/     可回测特征
data/attention/artifacts/    截图、导出文件、证据快照
data/attention/reports/      日报、周报、专题报告
```

生产数据不应直接提交进 Git 仓库。Git 保存契约、样例、配置、口径、任务和小型测试夹具；真实数据进入受控数据存储，并在仓库中登记工件引用和哈希。

## 9. 特征体系

### 9.1 单平台基础特征

- `level_percentile_30d`：近 30 日水平分位数
- `change_1d`、`change_7d`、`change_30d`
- `acceleration_7d`：增速的变化
- `persistence_days`：连续上升或高位天数
- `spike_score`：相对历史分布的异常程度
- `reversal_score`：峰值后的衰减强度
- `concentration_score`：热度是否集中在少数头部内容
- `organic_score`：自然扩散与营销投放的初步区分

### 9.2 跨平台特征

- `active_search_breadth`：主动搜索上涨的平台数量
- `content_supply_breadth`：内容供给上涨的平台数量
- `cross_platform_consistency`：方向一致性
- `search_to_content_lead_days`：搜索相对内容供给的领先天数
- `professional_to_search_lead_days`：专业信号相对大众搜索的领先天数
- `attention_to_market_lead_days`：注意力相对资本市场的领先天数
- `platform_sequence_pattern`：平台扩散顺序
- `narrative_divergence`：平台内容观点分歧

### 9.3 组合评分

首版不固定一套永久权重。先保留可解释分项：

```text
ASE = f(
  ActiveSearchDemand,
  ContentSupplyMomentum,
  DiffusionMomentum,
  ProfessionalConfirmation,
  InstitutionalConfirmation,
  MarketSynchronization,
  DataQualityPenalty,
  ManipulationRiskPenalty
)
```

在样本不足时使用透明规则和等权基线。积累足够历史后，再按主题类型、平台和市场状态学习权重。任何模型评分必须同时展示分项、证据和质量惩罚。

## 10. 领先滞后与因果边界

系统可以检验时间领先关系，但不能把相关性自动解释为因果。研究至少包括：

1. 交叉相关与滚动领先天数。
2. 事件窗口研究。
3. 不同市场状态下的分组检验。
4. 主题间对照和安慰剂测试。
5. 关键词词包版本敏感性。
6. 平台算法改版前后结构突变检测。
7. 营销投放、重大新闻和公司公告的混杂控制。

结论必须区分：描述性、预测性、因果候选和不可判断。

以“世界模型”为第一个主题包，建立从数据源验证到手工日报、再到自动化回测的完整最小闭环。

- 空间智能
- 具身智能世界模型
- 机器人世界模型
- 视频世界模型

公司名、产品名和模型名放在独立子主题，不与核心概念词直接求和。

### 11.3 首批观察

- 百度指数的搜索、资讯、媒体三类指标

## 13. 实施路线图

### Phase 0：治理与事实盘点

- 建立主题、来源、指标和工件契约。
- 把所有候选源登记到 Capability Registry。
- 把未知权限、数据口径和合规问题登记到 Unknown Registry。
- 固化 `NO_TRADE` 和研究侧车边界。
- 建立词包版本规则和数据来源引用规则。

### Phase 1：人工 MVP

- 只选“世界模型”一个主题。
- 连续采集一段时间的核心平台数据。
- 允许人工录入，但必须附证据和时间戳。
- 生成第一版每日表和每周阶段判断。
- 不做自动交易，不做复杂机器学习。

### Phase 2：半自动采集与质量门禁

- 优先接入官方 API、官方导出和稳定公开页面。
- 建立失败重试、延迟、缺失、口径漂移监控。
- 自动生成数据质量报告。
- 对登录、验证码和商业权限源保留人工流程。

### Phase 3：特征与看板

- 建立单平台和跨平台特征。
- 展示主动搜索、内容供给、传播消费、专业确认和市场同步的分层曲线。
- 输出扩散阶段、置信度和证据。

### Phase 4：历史回测

- 选择多个技术主题建立对照组。
- 检验搜索是否稳定领先内容和市场。
- 评估不同主题和市场状态的差异。
- 输出失败案例和不可交易结论，避免只展示成功案例。

### Phase 5：研究系统在线化

- 形成每日增量任务和异常提醒。
- 向事件情报模块和第二大脑输出只读研究工件。
- 通过确定性门禁向策略研究层提供特征。
- 保持真实下单链路独立。

## 14. 验收标准

第一版工程验收至少满足：

1. 任意特征可追溯到原始观察和来源工件。
2. 不同平台指标没有被错误解释为同一口径。
3. 缺失值、失败值和零值严格区分。
4. 词包和公司映射均可版本化。
5. 数据源权限状态明确，没有伪造的“自动接口”。
6. 能输出主动搜索与内容扩散的领先滞后结果。
7. 能展示至少一个反例或失败主题。
8. 所有研究输出默认 `NO_TRADE`。
9. 第二大脑只读接入，不反向污染交易事实层。
10. Codex、WorkBuddy 和 GPT 使用同一仓库文档与契约。

## 15. 关键风险

- 平台指数算法不透明或发生改版。
- 搜索词歧义导致虚假增长。
- 营销活动、刷量或平台推荐造成异常。
- 商业账号与普通账号看到的数据不同。
- 历史数据不可回溯，导致样本偏差。
- 技术热度与 A 股公司实际收入脱节。
- 先看价格再调整主题词包，造成未来函数。
- 多平台重复内容造成虚假的扩散宽度。
- 研究信号被误用为直接交易信号。

## 16. 需要持续登记的未知项

- 微信指数可获得的历史长度、导出能力和账号限制。
- 抖音指数升级后的稳定入口、分项口径和下载方式。
- 小红书搜索指数的账号权限和官方 API 范围。
- 微指数移动版的可记录字段和历史范围。
- B站 Z 指数的开放范围、历史数据和使用条款。
- 各平台的自动化访问边界。
- 券商研报授权和全文使用范围。
- A 股主题映射的人工审核责任人。

这些问题未验证前，状态保持 `UNKNOWN`，不得静默假设已经解决。

## 17. 决策记录

### ADR-ASE-001：主动搜索和内容热度分层

决定：主动搜索、内容供给、传播消费分别建模，不合成一个无法解释的热度数字。

原因：主动搜索代表更强意图，内容播放可能来自被动推荐。两者在传播链路中的位置不同。

### ADR-ASE-002：研究侧车，不进入订单热路径

决定：ASE 只提供研究特征和证据，不直接下单。

原因：平台数据口径、访问稳定性和叙事噪声都不足以承担订单关键路径。

### ADR-ASE-003：先做世界模型人工 MVP

决定：先用一个主题打通数据、证据、特征、阶段和回测闭环，再扩展到其他产业主题。

原因：先验证方法，避免一开始采集几十个平台和几百个主题，形成昂贵但不可验收的数据沼泽。
