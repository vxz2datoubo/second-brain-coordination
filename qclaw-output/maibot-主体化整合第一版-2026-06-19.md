# MaiBot 主体化整合第一版

日期: 2026-06-19

## 结论

以 `MaiBot` 作为整个系统的软件主体是当前工作区里最自然、成本最低、扩展性最好的路线。

它已经具备三项关键能力：

1. 自身可直连大模型，负责对话、规划、工具调用。
2. 自身已有 `A_Memorix` 长期记忆体系。
3. 自身已有 `MCP` 运行时，适合挂接第二大脑、Codex/WorkBuddy 和其他外部软件。

因此推荐将系统正式定义为：

`MaiBot 主体 + Live2D 表达层 + A_Memorix 宿主长期记忆 + 第二大脑外部知识库 + 决策系统前置审查 + Codex/WorkBuddy 执行层`

## 分层职责

### 1. MaiBot

- 系统主体
- 对话入口
- 工具编排器
- 会话上下文持有者
- 与 Live2D 的前端宿主

### 2. Live2D

- 桌面女友表现层
- 负责视觉、动作、情绪表达
- 不负责记忆、知识检索和执行逻辑

### 3. A_Memorix

- MaiBot 宿主内长期记忆
- 保存人物画像、聊天摘要、跨轮记忆、Episode、关系信息
- 适合承接“陪伴型”“连续关系型”记忆

### 4. 第二大脑

- 外部结构化知识库
- 保存高价值知识、经验、规则、教训、专题研究
- 适合承接“可复用知识”和“可检索决策依据”

### 5. 决策系统

- 在高价值动作前先查规则和历史教训
- 避免 MaiBot 在执行前直接冲出去
- 适合拦截：
  - 调工具
  - 改代码
  - 调 WorkBuddy/Codex
  - 做高风险外部操作

### 6. Codex / WorkBuddy

- 执行层
- 用于复杂多步任务、代码修改、命令执行、审查、自动化工作
- 不作为主体，只作为被 MaiBot 调用的能力层

## 推荐调用链

### 普通陪伴对话

用户 -> MaiBot -> 模型回复 -> Live2D 表达 -> A_Memorix 写回

### 知识型请求

用户 -> MaiBot -> second-brain MCP (`brain_search` / `brain_read`) -> 模型整合 -> 回复

### 决策型请求

用户 -> MaiBot -> second-brain MCP (`brain_consult`) / `before_decision()` -> 模型评估 -> 决定是否执行

### 执行型请求

用户 -> MaiBot -> 决策前门 -> codex-bridge MCP -> Codex/WorkBuddy 执行 -> 结果回传 -> 第二大脑沉淀 -> A_Memorix 记录会话影响

## 记忆分工建议

### A_Memorix 负责

- 用户偏好
- 关系状态
- 对话上下文
- 长期陪伴中的人物事实
- 聊天摘要与 episode

### 第二大脑负责

- AI视频/影视导演/A股/系统设计等专题知识
- 决策教训与红线
- 结构化方法论
- 可跨任务复用的稳定经验
- Codex / WorkBuddy 产出的高价值结果摘要

## 下一步建议

### P1

在 `MaiBot` 的工具执行前加入 decision gate：

- 低风险查询可直接调用
- 高价值执行先跑 `brain_consult`
- 高风险动作强制确认

### P2

定义 `MaiBot -> 第二大脑 -> A_Memorix` 的双记忆写入策略：

- 对话人格记忆优先写 `A_Memorix`
- 稳定知识和方法论优先写第二大脑
- 执行结果按重要性双写

### P3

把 Live2D 状态与内部决策态打通：

- 情绪
- 忙碌状态
- 是否在执行任务
- 是否命中风险提醒

### P4

如果后面需要“真正调 WorkBuddy 而不是只调 Codex”，再补一层专用 WorkBuddy MCP bridge。
当前第一版可以先以 `codex-bridge` 承担执行层角色。

## 重启提示

本次改动后，需要重启 `MaiBot` 才会重新加载新增的 MCP server 配置。
