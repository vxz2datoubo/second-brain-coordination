## 一期总目标

让 `MaiBot` 具备：

1. 内置知识脑最小可用版
2. 决策脑主链路
3. 任务账本最小版

5. WorkBuddy / Codex 执行接入
6. 任务完成提醒闭环

## 阶段拆分

### 阶段 P0：边界固化

目标：

- 固化 `A_Memorix / InnerBrainStore / TaskLedger` 三分结构
- 固化 `brain_*` 统一接口
- 固化 Decision Brain 的入口地位

验收：

- 新增任何记忆写入时，都能说明属于哪一类
- 复杂任务不再直接跳进执行器

### 阶段 P1：InnerBrainStore 最小可用版

目标：

- 建立内置知识脑最小数据结构
- 支持存储 / 检索 / 咨询 / 读节点 / 导出

验收：

- 可调用 `brain_search`
- 可调用 `brain_consult`
- 可调用 `brain_digest`
- 可调用 `brain_read`
- 预留 `brain_export`

### 阶段 P2：TaskLedger 最小可用版

目标：

- 给复杂任务一个正式账本
- 记录任务计划、状态、调用链、结果、提醒状态

验收：

- 复杂任务可追踪
- 能看出任务是否完成
- 能知道是否已提醒用户

### 阶段 P3：QClaw / WorkBuddy 委派

目标：

- QClaw 只做重算分析
- WorkBuddy / Codex 只做执行
- 决策层决定是否调用

验收：

- 长文本分析能走 QClaw
- 文件 / 代码 / 自动化能走 WorkBuddy / Codex
- 没有越层调用

### 阶段 P4：提醒闭环

目标：

- 任务完成后让 MaiBot 统一提醒用户
- 可通过 Live2D / QQ 回推

验收：

- 任务完成状态写入 TaskLedger
- Reminder Center 能读取并触发提醒

## WorkBuddy 任务清单

### WB-01

任务名：建立 `InnerBrainStore` 最小模块骨架

要求：

- 明确存储路径
- 明确最小数据模型
- 明确 `brain_search / brain_digest / brain_read / brain_consult / brain_export` 接口位置
- 不要和 A_Memorix 混用存储

产出：

- 模块骨架代码
- 数据模型说明

### WB-02

任务名：建立 `TaskLedger` 最小模块骨架

要求：

- 至少支持任务创建、状态更新、结果写入、提醒标记
- 能关联工具调用记录

产出：

- TaskLedger 代码
- 状态模型说明

### WB-03

任务名：把复杂任务执行正式挂到 `Task Orchestrator`

要求：

- 不允许直接从对话链跳执行器
- 必须先有任务记录
- 必须先经过 Decision Brain

产出：

- Orchestrator 调用链实现
- 调用时序说明

### WB-04

任务名：建立 `Reminder Center` 最小闭环

要求：

- 任务完成后能记录为“待提醒”
- 能转成 MaiBot 主体的提醒事件

产出：

- Reminder Center 最小实现
- 与 TaskLedger 的衔接

## QClaw 任务清单

### QC-01

任务名：为 `InnerBrainStore` 设计最小知识模型

要求：

- 只做分析方案
- 输出节点字段建议、关系字段建议、标签策略
- 区分“知识 / 规则 / 教训 / 复盘”

产出：

- 结构设计草案

### QC-02

任务名：分析 `brain_consult` 的最佳最小实现

要求：

- 只做重算分析
- 给出决策前检索最小流程
- 给出如何避免噪声命中

产出：

- 决策咨询规则草案

### QC-03

任务名：分析“哪些结果值得沉淀进内置知识脑”

要求：

- 给出高价值结果判定标准
- 区分“只进 A_Memorix”与“应进 InnerBrainStore”

产出：

- 沉淀规则草案

### QC-04

任务名：分析未来从内置知识脑迁移到外挂第二大脑的清洗策略

要求：

- 给出迁移时的数据分类标准
- 指出哪些噪声应丢弃

产出：

- 未来迁移清洗规则

要求：

- 看是否引入第二主体
- 看是否混存

要求：
