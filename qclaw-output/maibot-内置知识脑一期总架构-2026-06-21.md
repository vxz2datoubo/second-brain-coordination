# MaiBot 内置知识脑一期总架构

日期：2026-06-21
状态：定版建议
适用范围：MaiBot / WorkBuddy / QClaw / Codex 协作开发

## 定版结论

当前阶段采用：

`MaiBot 主体 + 内置知识脑一期 + A_Memorix 关系记忆 + 外部执行工具`

暂不把“外挂第二大脑知识库”作为主库。
但从第一天开始，内置知识脑必须按“未来可外迁”为前提设计。

也就是说：

1. 现在先内置，降低系统复杂度
2. 不做双主库
3. 不做双向同步
4. 以后如果 MaiBot 扛不住，再把“内置知识脑”迁移为外挂第二大脑

---

## 为什么这一版最适合现在

### 适合你的原因

1. 你现在最优先的是把 `MaiBot` 主体真正做起来
2. 你暂时还不确定 `MaiBot` 的能力边界
3. 你不想现在就维护两套脑子
4. 你以后仍然要做高精度复杂任务，所以不能把后路堵死

### 这版的核心优势

1. 单体简单
2. 主体集中
3. 数据只进一处
4. 后续可迁移
5. 仍能接入 WorkBuddy / QClaw / Codex

---

## 一句话架构

```text
用户
├─ QQ
├─ 微信（未来）
├─ Live2D 桌宠
└─ 通知 / 提醒
        ↓
      MaiBot Kernel
        ↓
├─ Companion Brain
├─ Decision Brain
├─ Inner Brain
├─ Memory Router
├─ Task Orchestrator
└─ Reminder Center
        ↓
├─ A_Memorix
├─ Inner Brain Store
└─ Task Ledger
        ↓
├─ QClaw
├─ WorkBuddy / Codex
└─ Other MCP Tools
```

## 三套核心存储

### 1. A_Memorix

只负责“人和关系”

存：

- 用户偏好
- 人物画像
- 关系状态
- 聊天摘要
- Episode
- 情绪余波
- 关系温度

不负责：

- 决策教训主库
- 专题知识主库
- 跨任务方法论主库

### 2. Inner Brain Store

这是“内置知识脑一期”的主库。

只负责“知识与决策依据”

存：

- 专题知识
- 规则
- 教训
- 红线
- 任务复盘
- 高价值执行结论
- 可复用方法论

不负责：

- 用户关系记忆
- 聊天陪伴型记忆
- 任务运行状态

### 3. Task Ledger

只负责“工作过程和结果”

存：

- 任务 ID
- 任务分类
- 计划
- 调用记录
- 中间结果
- 最终结果
- 完成状态
- 提醒状态

## 五个脑的职责

### 1. Companion Brain

负责：

- 人设稳定
- 回复风格
- 情绪连续性
- 关系表现
- Live2D 情绪映射
- 主动补充 / 延迟反刍

### 2. Decision Brain

负责：

- 判断当前请求类型
- 判断是否需要查知识脑
- 判断是否需要 QClaw
- 判断是否需要 WorkBuddy / Codex
- 判断风险
- 判断是否允许执行

### 3. Inner Brain

负责：

- 知识存储
- 教训检索
- 规则检索
- 高价值结论沉淀
- 给 Decision Brain 提供依据

### 4. Memory Router

负责把数据分流：

- 关系信息 → A_Memorix
- 知识信息 → Inner Brain Store
- 任务过程 → Task Ledger

这层以后决定迁移难度，必须先做对。

### 5. Task Orchestrator

负责：

- 多步复杂任务
- 调用 QClaw
- 调用 WorkBuddy / Codex
- 状态跟踪
- 结果回执

### 6. Reminder Center

负责：

- 任务完成提醒
- 超时提醒
- 回访提醒
- 通过 QQ / Live2D / 微信回推

---

## 请求处理流程

### A. 普通陪伴对话

用户消息
→ Companion Brain
→ 读取 A_Memorix
→ 回复
→ 写回 A_Memorix

### B. 知识型请求

用户消息
→ Decision Brain 判断为知识请求
→ 检索 Inner Brain Store
→ Companion Brain 整合输出
→ 必要时沉淀新知识

### C. 高精度复杂任务

用户消息
→ Decision Brain
→ 查 Inner Brain Store 中的规则 / 教训 / 复盘
→ 生成任务计划
→ Task Orchestrator
→ QClaw 做重算分析
→ WorkBuddy / Codex 做执行
→ 结果进入 Task Ledger
→ 高价值结论沉淀到 Inner Brain Store
→ Reminder Center 提醒用户

## 为什么不做双主库

当前阶段不推荐：

- 内置知识脑一份
- 外挂第二大脑再来一份
- 两边同时读写
- 两边互相同步

原因：

1. 数据归属会乱
2. 决策时不知道该查哪边
3. 投喂会重复劳动
4. 后期同步冲突会很多
5. 当前阶段收益低于复杂度

所以一期规则很明确：

`只有一个主知识库，就是 MaiBot 内置知识脑`

---

## 未来外迁策略

如果后面判断 `MaiBot` 的内置知识脑不够用了，不是简单复制数据库，而是走“迁移重建”：

### 第一步：导出

从内置知识脑导出结构化数据：

- 节点
- 关系
- 标签
- 类型
- 来源
- 时间

### 第二步：清洗

分出：

- 保留在 A_Memorix 的关系记忆
- 应迁移到外挂第二大脑的知识内容
- 应丢弃的低价值噪声

### 第三步：重新消化

把清洗后的知识按新规则重新写入外挂第二大脑。

### 第四步：切换

上层接口保持不变，只把 `brain_*` 的实现从“本地模块”切到“外挂服务”。

---

## 现在就必须遵守的接口约束

即便一期先内置，也必须统一这些抽象接口：

- `brain_search(query, top_k, scope)`
- `brain_consult(topic, risk_level)`
- `brain_digest(text, title, source, kind)`
- `brain_read(node_id, relation_depth)`
- `brain_export(...)`

注意：

现在这些接口可以先由 `MaiBot` 内部模块实现。
以后外挂时，只替换实现，不替换调用方式。

## QClaw 的定位

QClaw 在这套架构里只负责：

- 长文本总结
- 多候选方案
- 风格润色
- 复杂分析
- 高消耗文本推理

QClaw 不负责：

- 主体人格
- 最终决策拍板
- 文件事实判断
- 真实执行修改

---

## WorkBuddy / Codex 的定位

WorkBuddy / Codex 在这套架构里只负责：

- 代码修改
- 文件操作
- 自动化流程
- 复杂任务执行
- 落地外部工具链

它们不是知识主库，也不是人格主体。

## 一期开发顺序

### P0

- 固化主体架构
- 明确三套存储边界
- 把 Decision Brain 放进主链路

### P1

- 定义 Inner Brain Store 的最小数据模型
- 建立 `brain_*` 抽象接口
- 确定 Memory Router

### P2

- 打通 QClaw 委派
- 打通 WorkBuddy / Codex 执行
- 建立 Task Ledger

### P3

- 做 Reminder Center
- 做任务完成提醒
- 做 Live2D / QQ 状态回推

### P4

- 设计 Brain Export
- 设计未来外迁脚本
- 设计知识清洗和重建流程

---

## 一期验收标准

满足以下条件，说明一期架构做对了：

1. 用户视角只有一个 MaiBot
2. 关系记忆和知识记忆严格分开
3. 复杂任务先经过决策，再调用工具
4. QClaw 和 WorkBuddy 都只是能力层
5. 高价值执行结果能沉淀进内置知识脑
6. 将来可以把内置知识脑导出重建成外挂第二大脑

---

## 当前建议

现在正式采用这条路线：

`先做 MaiBot 内置知识脑一期`

不是临时凑合，而是一个明确可持续、可迁移、可扩展的第一阶段架构。
