# MaiBot 单体多脑架构蓝图

日期：2026-06-21
状态：推荐作为后续统一总架构

## 一句话结论

从零开始时，推荐采用：

`MaiBot 单体主程序 + 内部多脑分层 + 外部执行工具接入`

不是“很多松散外部服务拼起来”，也不是“把所有能力揉成一条聊天 prompt”。

产品上只暴露一个主体：`MaiBot`

架构上拆成 5 个脑：

1. 陪伴脑
2. 决策脑
3. 关系记忆脑
4. 知识脑
5. 执行脑

---

## 先回答关键问题

### A_Memorix 能不能替代外部第二大脑

结论：`不能完全替代，只能替代其中一部分。`

`A_Memorix` 适合承担：

- 用户偏好
- 人物画像
- 关系状态
- 聊天摘要
- Episode
- 陪伴记忆

但它不适合直接完整替代外部第二大脑目前承担的这些职责：

- 决策教训库
- 跨项目知识复用
- 可独立服务的知识图谱
- 对 WorkBuddy / Codex / QClaw 的统一知识出口
- 知识健康 / 自校验 / 演化 / 事件 / 审核统计

所以推荐结论不是“二选一”，而是：

- `A_Memorix = 关系型长期记忆`
- `BrainStore = 外部第二大脑的能力内化为 MaiBot 内部知识脑`

也就是说：

如果从零开始，可以不再保留“外部独立 HTTP 第二大脑服务”的部署形态；
但不能不要“第二大脑这类能力”本身。

它应该以内置模块形式保留在 `MaiBot` 里，而不是被 `A_Memorix` 吞掉。

## 能力对比

| 维度 | A_Memorix | 外部第二大脑现有能力 | 是否可直接由 A_Memorix 替代 |
|------|-----------|----------------------|------------------------------|
| 用户偏好 / 人物画像 | 强 | 弱 | 可以 |
| 聊天摘要 / Episode | 强 | 一般 | 可以 |
| 陪伴关系连续性 | 强 | 弱 | 可以 |
| 专题知识库存储 | 一般 | 强 | 不建议 |
| 决策教训 / 红线规则 | 弱到中 | 强 | 不建议 |
| 跨系统统一检索出口 | 弱 | 强 | 不建议 |

| 自校验 / 知识健康 / 演化 | 弱 | 强 | 不能完整替代 |
| WorkBuddy / Codex / QClaw 共用知识源 | 弱 | 强 | 不建议 |

## 各层职责

### 1. Companion Brain

负责：

- 人设稳定
- 情绪连续性
- 关系温度
- 回复风格
- Live2D 状态映射
- 主动补充与延迟反刍

输入：

- 当前消息
- 会话上下文
- A_Memorix 关系记忆
- Decision Brain 给出的表达约束

输出：

- 可见回复
- Live2D 情绪 / 动作状态
- 关系更新信号

### 2. Decision Brain

负责：

- 当前请求分类
- 风险判断
- 是否需要查教训
- 是否调用外部工具
- 是否需要 QClaw 重算
- 是否需要 WorkBuddy / Codex 执行

输入：

- 用户请求
- 会话状态
- BrainStore 检索结果
- 历史执行记录

输出：

- 决策结果
- 工具调用计划
- 风险提示

### 3. Memory Router

负责把信息分流到正确的存储：

- 陪伴关系信息 → `A_Memorix`
- 可复用知识 / 教训 / 方法论 → `BrainStore`
- 执行过程 / 状态 / 结果 → `Task Ledger`

这是整个系统后期不串味的关键层。

### 4. Knowledge Brain

这是“第二大脑能力”的内化版本。

负责：

- 专题知识存储
- 规则 / 教训存储
- 决策前检索
- 高价值结果沉淀
- 跨任务复用
- 统一知识出口

注意：

这里的推荐是把“第二大脑能力”内化进 `MaiBot`，不是删掉这层能力。

### 5. Task Orchestrator

负责：

- 多步任务计划
- 调用 QClaw
- 调用 WorkBuddy / Codex
- 状态跟踪
- 完成回执

它不是人格层，而是工作层。

### 6. Reminder / Notification Center

负责：

- 任务完成提醒
- 超时提醒
- 主动回访
- 根据场景决定通过 Live2D、QQ、微信哪个通道提醒

---

## 三套存储必须分开

### A_Memorix Store

只存：

- 偏好
- 关系
- 人物画像
- 聊天摘要
- Episode
- 情绪余波

### BrainStore

只存：

- 专题知识
- 规则
- 教训
- 决策红线
- 高价值执行复盘
- 跨任务方法论

### Task Ledger

只存：

- 任务计划
- 状态
- 工具调用
- 中间结果
- 最终结果
- 提醒状态

三者严禁混成一张总记忆表。

---

## 推荐的数据流

### 普通陪伴对话

用户消息
→ Companion Brain
→ 读取 A_Memorix
→ Decision Brain 判断是否仅需普通回复
→ 回复
→ 写回 A_Memorix

### 知识问答

用户消息
→ Decision Brain 判断为知识型请求
→ 检索 BrainStore
→ Companion Brain 整合输出
→ 必要时写回新知识

### 高精度复杂任务

用户消息
→ Decision Brain
→ 查 BrainStore 教训 / 红线
→ 生成执行计划
→ Task Orchestrator
→ QClaw 重算分析
→ WorkBuddy / Codex 执行
→ 结果写入 Task Ledger
→ 高价值结论沉淀到 BrainStore
→ MaiBot 提醒用户

---

## QClaw 和 WorkBuddy 的定位

### QClaw

只负责重算：

- 长文本总结
- 多方案生成
- 风格润色
- 复杂分析
- 批量关系样本评审

它不是主体，不负责最终拍板。

### WorkBuddy / Codex

只负责执行：

- 改代码
- 调工具
- 文件操作
- 自动化流程
- 多步工作流落地

它们不是人格主体，不直接决定陪伴风格。

---

## 为什么这个架构适合以后做复杂工作

因为它同时满足：

1. 单体简单
   - 一个主体
   - 一个入口
   - 一个主控制台

2. 能做复杂任务
   - 决策层
   - 知识层
   - 执行层
   - 重算层

3. 不容易后期失控
   - 记忆分层
   - 职责分层
   - 工具分层

---

## 绝对约束

1. `MaiBot` 永远是唯一主体
2. `Live2D` 是表现层，不是主逻辑中枢
3. `QClaw` 不是人格主体，只是重算副脑
4. `WorkBuddy / Codex` 不是主体，只是执行层
5. `A_Memorix` 不能直接吞并全部第二大脑能力
6. `BrainStore` 与 `A_Memorix` 必须长期共存
7. 决策必须先于复杂执行
8. 高价值执行结果必须可沉淀、可复用、可审查

---

## 从零开始的实现优先级

### P0

- 定义 `MaiBot Kernel`
- 明确三套存储边界
- 把 Decision Brain 放进主链路

### P1

- 做 A_Memorix 关系记忆接入
- 做 BrainStore 最小版
- 做 Task Ledger

### P2

- 接入 QClaw
- 接入 WorkBuddy / Codex
- 做任务完成提醒

### P3

- 接入 Live2D 状态联动
- 接入 QQ / 微信双通道
- 做复杂任务编排优化

### P4

- 知识健康
- 规则演化
- 自动复盘
- 决策质量评估

---

## 最终建议

如果今天从零开始：

推荐不是“保留外部第二大脑服务不动”，
也不是“全部只靠 A_Memorix”。

推荐路线是：

`把第二大脑的核心能力内化成 MaiBot 内部的 Knowledge Brain / BrainStore 模块`

这样既保留单体简单度，
又保留未来做复杂思维与高精度工作的能力上限。
