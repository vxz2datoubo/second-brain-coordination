## 三、逐字稿获取方案（核心难点）

抖音视频逐字稿是系统最关键也最难获取的数据。以下是 **4种方案对比**：

### 方案A：DOM字幕提取 ⭐⭐⭐⭐⭐ (首选)

```
原理: 抖音Web版部分视频在播放器内有 <span class="caption"> 或类似元素显示AI字幕
实施:
  1. Playwright打开视频页，等待播放器加载
  2. 执行JS监听播放器DOM变化: MutationObserver监听字幕容器
  3. 持续采集整个播放过程的字幕文本
  4. 去重+时间戳对齐 → 完整逐字稿

优点: 零额外依赖，速度快，准确率高(官方字幕)
缺点: 非全部视频都有AI字幕；字幕容器class名可能变化需维护
```

**关键代码思路：**
```javascript
// 在Playwright页面注入
const captions = [];
const observer = new MutationObserver((mutations) => {
  for (const m of mutations) {
    for (const node of m.addedNodes) {
      // 匹配字幕容器 (需实测确定选择器)
      if (node.className?.includes('caption') || 
          node.className?.includes('subtitle')) {
        captions.push({ time: Date.now(), text: node.textContent });
      }
    }
  }
});
observer.observe(document.body, { childList: true, subtree: true });
```

### 方案B：音频下载 + Whisper转写 ⭐⭐⭐⭐ (最可靠)

```
原理: 下载视频音频流 → 本地Whisper模型转写
实施:
  1. 获取视频直链 (需解析抖音分享链接 → 无水印视频地址)
  2. 下载音频轨道 (ffmpeg分离音频: ffmpeg -i video.mp4 -vn audio.wav)
  3. faster-whisper 转写 (本地GPU加速)
  4. 输出带时间戳的SRT/VTT逐字稿

优点: 100%覆盖任何视频，质量可控
缺点: 需要下载视频(数MB~数十MB)，转写耗时(1分钟视频约5-15秒)
      需要本地部署Whisper模型 (~1.5GB)
```

**所需工具链：**
- `faster-whisper` (Python, GPU加速)
- `ffmpeg` (音频提取)
- 抖音无水印解析 (已有第三方API或xposed方案)

### 方案C：分享链接API ⭐⭐⭐

```
原理: 通过抖音分享接口获取视频元数据(部分视频包含文案全文)
实施:
  1. 解析分享短链接 → 获取video_id
  2. 调用抖音item接口 (需模拟客户端签名)
  3. desc字段通常包含完整文案(不是逐字稿，但是创作者写的脚本)

优点: 零视频下载，API直接返回文本
缺点: desc≠逐字稿，很多创作者只写简略文案；
      需要客户端签名算法(较复杂)
```

### 方案D：手动粘贴 ⭐⭐ (兜底)

```
原理: 用户手动复制视频文案/字幕粘贴
适用: 以上方案均失败时的终极保底
```

## 四、数据流详解

```
输入: 抖音视频链接 (如 https://www.douyin.com/video/xxxxx)
  │
  ├─[Step 1] Playwright打开页面 → 等待加载 → 处理登录弹窗
  │   ├── 提取: title, author, publish_time, stats(like/comment/share)
  │   ├── 提取: desc文案, tags, @mentions
  │   ├── 提取: 逐字稿 (方案A → 失败则方案B)
  │   ├── 截图: 视频首帧 → screenshots/YYYY-MM-DD/video_id_cover.png
  │   ├── 截图: 评论区 → screenshots/YYYY-MM-DD/video_id_comments.png
  │   └── 采集: 评论列表 (滚动加载→提取前N条/热门+N条最新)
  │
  ├─[Step 2] QClaw分析引擎
  │   ├── 文案纠错分析:
  │   │   · 错别字/拼音错误扫描
  │   │   · 语法结构分析
  │   │   · 事实性错误检测
  │   │   · 逻辑漏洞识别
  │   │   · 夸大/误导表述标注
  │   │
  │   ├── 评论洞察:
  │   │   · 情感极性分布 (正面/负面/中性)
  │   │   · 高频关键词提取
  │   │   · 质疑型评论识别 (fact-check信号)
  │   │   · 补充型评论提取 (用户贡献增量信息)
  │   │   · 用户画像概览 (粉丝/路人/黑粉比例)
  │   │
  │   ├── 知识库交叉验证:
  │   │   · 检索117节点中相关知识
  │   │   · 匹配相似主题/事件
  │   │   · 标注创作者已知知识盲区
  │   │   · 输出知识补充建议
  │   │
  │   └── 综合研判:
  │       · 内容可信度评分 (0-100)
  │       · 核心发现摘要 (3-5条)
  │       · 值得关注的高价值评论
  │       · 知识库更新建议
  │
  └─[Step 3] 输出
      ├── 生成 Markdown 报告 → F:\ai\qclaw-output\抖音分析_YYYY-MM-DD_视频标题.md
      ├── 关键发现入库第二大脑 → POST /api/digest/text
      └── 截图归档 → D:\抖音分析\screenshots\YYYY-MM-DD\
```

## 七、关键技术决策

| 决策点 | 推荐 | 理由 |
|--------|------|------|
| Playwright vs API | Playwright | 抖音API签名复杂且不稳定，DOM提取更可控 |
| 逐字稿方案 | A优先→B兜底 | DOM提取零成本，Whisper备选全覆盖 |
| 评论采集深度 | 热门50条+最新50条 | 平衡分析质量与采集耗时 |
| QClaw分析模式 | 子Agent独立运行 | 分析任务重，独立Session不阻塞主会话 |
| 知识库交互 | 检索注入+事后入库 | 分析时检索增强，结论后入库沉淀 |

## 八、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 抖音反爬升级 | 无法获取页面 | 维护登录Cookie池，控制请求频率 |
| DOM结构变化 | 选择器失效 | 属性选择器而非class名，定期回归测试 |
| 视频无AI字幕 | 方案A失效 | 自动fallback到方案B Whisper转写 |
| 视频下载失败 | 方案B失败 | 降级为desc文案分析(跳过逐字稿) |
| 评论接口限流 | 采集不全 | 分段采集+随机延迟，不追求全量 |
