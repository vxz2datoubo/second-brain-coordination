# MaiBot 协作代理接力指令

日期：2026-06-21
适用对象：WorkBuddy / QClaw / Codex / 后续协作代理

## 目的

本文件用于保证后续协作代理继续开发时，不偏离 `MaiBot 单体多脑架构蓝图`。

所有协作代理在继续工作前，都应先读取：

1. `F:/ai/qclaw-output/maibot-单体多脑架构蓝图-2026-06-21.md`
2. 本文件

---

## 最高优先级原则

1. `MaiBot` 是唯一主体
2. 所有新增能力都必须挂在 `MaiBot` 之下
3. 不允许把 `QClaw`、`WorkBuddy`、`Codex` 做成新的主脑
4. 不允许把 `Live2D` 设计成中枢逻辑层
5. 不允许把关系记忆、知识记忆、任务记忆混成一个存储
6. 不允许让复杂执行绕过决策层
7. 所有高价值结果必须能沉淀

---

## 各代理分工

### WorkBuddy

职责：

- 复杂任务执行
- 文件与代码修改
- 自动化流程落地
- 任务编排实现
- 工具连接器实现

禁止：

- 修改主体架构方向
- 把自己抬升为主体
- 把人格逻辑塞进执行链

### QClaw

职责：

- 重算分析
- 长文本总结
- 多候选方案
- 风格润色
- 复杂逻辑分析草案

禁止：

- 直接假定文件事实
- 代替决策层拍板
- 代替执行层做真实修改
- 代替 MaiBot 生成人格总控

### Codex

职责：

- 代码改造
- 架构约束守门
- 边界收敛
- 验证 WorkBuddy / QClaw 输出是否符合总架构

---

## 后续开发只能围绕的三套存储

1. `A_Memorix`
   - 人物画像
   - 关系记忆
   - 偏好
   - 聊天摘要
   - Episode

2. `BrainStore`
   - 专题知识
   - 教训
   - 红线规则
   - 可复用方法论
   - 高价值任务复盘

3. `Task Ledger`
   - 任务状态
   - 计划
   - 调用记录
   - 结果
   - 提醒状态

任何新增存储设计，都必须先说明它属于这三者中的哪一类。

---

## 推荐继续开发的顺序

### 第一阶段

- 把 `Decision Brain` 在 `MaiBot` 主链路里彻底坐实
- 明确请求分类
- 明确复杂任务前的检索与审查门

### 第二阶段

- 设计 `BrainStore`
- 把原外部第二大脑的核心能力按模块迁入
- 但保留清晰 API 边界

### 第三阶段

- 建立 `Task Ledger`
- 将复杂任务状态、结果、提醒整合进去

### 第四阶段

- 优化 `QClaw` 委派规则
- 优化 `WorkBuddy / Codex` 执行规则

### 第五阶段

- 做 Live2D 联动
- 做任务完成提醒
- 做 QQ / 微信多通道回推

## 这份架构的验收标准

如果后续代理做出来的结果满足以下条件，就算没有跑偏：

1. 用户视角始终只有一个 `MaiBot`
2. 关系记忆没有和知识教训混存
3. 复杂任务能先审查、再执行
4. 执行结果能沉淀并提醒
5. `QClaw` 与 `WorkBuddy` 都是能力层，不是主体层

如果不满足，就说明已经偏离总架构。
