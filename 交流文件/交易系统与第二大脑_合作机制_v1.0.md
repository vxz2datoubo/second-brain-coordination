# 交易系统与第二大脑：企业级项目集集成决策与实施蓝图 v1.0

**状态：架构定版建议（未授权实盘）**  
**适用范围：A 股优先、本地优先、可由 Codex / WorkBuddy / Maibot / GPT 协作执行**  
**编制目的：回答“合成一个系统还是分开做、先后顺序是什么、怎样既独立又可联动”并给出可实施的项目集设计。**

---

## 0. 一页结论：采用“一个项目集、两个产品、一个受控联动层”

### 0.1 最终选择

不要做成一个运行时互相纠缠的“大系统”；也不要等交易系统全部完成后才第一次设计接口。

采用以下结构：

```text
企业级智能投资项目集（唯一项目章程、统一治理、统一契约）
│
├── 产品 A：A 股量化交易平台（独立可用、独立发布、独立停机）
│   ├── 数据 / 规则 / 回放 / 研究 / 策略 / 风控 / OMS / 对账
│   └── 交易关键路径：确定性服务，不依赖第二大脑或任一 LLM
│
├── 产品 B：第二大脑认知操作系统（独立可用、独立发布、独立降级）
│   ├── 记忆 / 知识证据 / 经验 / 资产 / 检索 / 规划 / 协作
│   └── 可服务多个领域，交易是其中最严格的专业域
│
└── 产品 C：受控联动层（唯一跨域入口，不是第三套业务真相）
    ├── Canonical Contracts + Schema Registry
    ├── Integration Gateway + Policy Engine + Audit / Provenance
    ├── 只读资源、候选写入、事件投影与任务交接
    └── 永不持有交易豁免、实盘密钥或订单路由权
```

一句话：**项目层面从第一天共同规划；代码与运行时层面保持可独立；能力层面通过版本化契约安全联动。**

### 0.2 对四个问题的直接回答

| 你的问题 | 结论 |
|---|---|
| 要不要整合成一个“系统文件”？ | 需要一个总项目章程和一份集成契约；不应把两个系统压成一个代码程序或一个共享数据库。文档是治理入口，真正的系统边界由契约、权限、工件、测试和部署定义。 |
| 交易系统能否单独使用？ | 必须能。第二大脑、向量库、图谱、Agent、LLM 全部失效时，交易系统仍能在既批准的本地数据、规则、风控、OMS 与人工授权下安全运行；否则降级为 `NO_TRADE`。 |
| 第二大脑能否调用交易系统？ | 可以，但先调用只读资源和受控研究工具；只能产出 `ResearchProposal` / `OrderIntentDraft`，不能越过规则、风控、人工审批和 OMS。 |
| 应先做哪个？ | **共建边界，交易优先落地。** 先完成共同基础与交易最小可信闭环；第二大脑同时只读接入并积累证据。不是先做完 A 再开始 B，也不是二者平均用力。 |

### 0.3 为什么不是“全部微服务化”或“完全延后对接”

已发布的工程经验表明，微服务提高独立演进能力的同时会引入服务依赖、排障和级联 QoS 风险；甚至有大型项目因权衡不利回到单体。[Google 对 Istio 的工业复盘](https://research.google/pubs/the-monolith-strikes-back-why-istio-migrated-from-microservices-to-a-monolithic-architecture/)  
因此首选**模块化单体 + 进程边界 + 契约测试**，只把确实存在独立伸缩、独立故障域、独立权限或独立发布需求的部分拆成服务。分布式追踪必须随架构进入，而不是以后补；[Dapper](https://research.google/pubs/dapper-a-large-scale-distributed-systems-tracing-infrastructure/) 的经验说明可追踪性会成为大型系统的基础能力。

| [PBO / CSCV 研究](https://scholarworks.wmich.edu/math_pubs/42/) | 金融回测的模型选择可产生过拟合，必须记录试验族与样本外表现。 | A-/同行评审研究 | 某个策略必然失效或必然赚钱。 | 第二大脑的经验不能直接升级为交易规律。 |

## 2. 两个系统的本体与不可交换边界

### 2.1 产品 A：交易平台的职责

交易平台是**金融控制系统**，其真相由可回放数据、规则版本、确定性计算、风险决策、订单状态和对账结果构成。

它负责：

- 交易日历、证券主数据、公司行动、A 股规则与点时数据；
- 数据质量、特征、回测、样本外验证、策略与组合；
- 独立硬风控、预交易校验、OMS、执行、成交/持仓/现金对账；
- 影子、纸面、受控实盘的逐级门禁、熔断与恢复；
- 研究结果的可复现性与交易结果的不可抵赖审计。

它**不负责**把所有项目知识都存进交易库，也不需要依赖自然语言 Agent 才能进行风控、下单、撤单或恢复。

### 2.2 产品 B：第二大脑的职责

第二大脑是**认知、知识与协作系统**，其真相由原始资产、来源、证据主张、记忆、决策记录、经验、版本和权限构成。

它负责：

- 用长期/情景/语义/程序/前瞻记忆保存项目与专业经验；
- 把论文、规则、公告、日志、复盘和文档组织为带来源、冲突、有效期、适用范围的知识；
- 为当前任务检索直接相关与“可能意外有帮助”的历史证据；
- 形成决策备选、反证、未知项、长期目标与研究任务；
- 服务交易以外的个人/项目/资产知识场景。

它**不负责**成为账户、订单、持仓、风险限额或市场事实的唯一真相源。

### 2.3 绝不合并的四项权威

| 权威对象 | 唯一权威 | 第二大脑可做什么 | 第二大脑不得做什么 |
|---|---|---|---|
| 市场事实、规则、持仓、订单、现金 | 交易平台的版本化账本与确定性服务 | 读取经授权的快照、解释、挂接证据 | 改写、补全、凭记忆纠错或直接作为交易事实。 |
| 生产密钥与券商会话 | Secrets Broker / 独立执行进程 | 不接触 | 读取、转发、缓存、出现在 Prompt、日志或知识库。 |
| 风控裁决 | 独立 Risk Engine | 说明风险原因、生成研究任务 | 赦免、覆盖、绕过或把草案变为订单。 |
| 用户的高风险决策 | 用户明确审批记录 | 提供已标注不确定性的选择 | 代替批准。 |

## 3. 推荐的三层集成架构

### 3.1 运行时拓扑

```text
                         ┌────────────────────────────────┐
                         │       人工审批与治理层          │
                         │ User / Policy / Audit / ADR     │
                         └──────────────┬─────────────────┘
                                        │
┌─────────────────────────────┐   ┌─────▼─────────────────────────┐
│ 第二大脑（独立部署）          │   │ 受控联动层：Gateway + Contracts │
│ Memory / Knowledge / Asset   │◄─►│ Auth / Schema / Policy / Trace  │
│ Retrieval / Planner          │   │ Event Projection / Handoff      │
└────────────┬────────────────┘   └─────┬─────────────────────────┘
             │                           │
             │ 可用性下降仅降级检索       │ 只读快照 / 候选提案 / 已审批事件
             │                           │
             │                       ┌───▼──────────────────────────┐
             │                       │ 交易平台（独立部署）           │
             │                       │ Data → Rules → Quality → Risk │
             │                       │ → OMS → Broker → Reconcile    │
             │                       └───┬──────────────────────────┘
             │                           │
             └──────不进入订单关键路径───┘
```

### 3.2 三种交互，不多不少

1. **交易 → 第二大脑：事件投影（优先、只读）**  
   将已脱敏并审批的数据快照、规则版本、实验、决策包、风险拒绝、回测结果、事故与复盘投影为知识候选。原始交易库不被第二大脑直接写入。

2. **第二大脑 → 交易：受控提案（有限写入）**  
   只提交 `ResearchProposal`、`EvidenceClaimDraft`、`ScenarioDraft`、`OrderIntentDraft`。后者仍是草案，不能直接触发券商请求。

3. **双向：可追溯引用（只传 ID，不复制真相）**  
   `DecisionPacket` 引用 `claim_id / experiment_id / asset_hash`；第二大脑引用 `dataset_snapshot_id / rule_snapshot_id / risk_decision_id`。任一侧需要详情时按权限回源读取。

### 3.3 强制的事件与契约

所有跨域事件至少包含：

```yaml
Envelope:
  event_id: uuid
  event_type: string
  producer: trading-platform | second-brain | gateway
  schema_version: semver
  trace_id: string
  idempotency_key: string
  as_of: timestamp                 # 当时可知的时间
  observed_at: timestamp           # 发生/采集的时间
  source_snapshot_id: string|null
  sensitivity: PUBLIC|INTERNAL|CONFIDENTIAL|TRADING_RESTRICTED
  provenance: {asset_hash: string, source_uri: string|null}
  policy_decision_id: string
  payload: object
```

**不可省略字段：** `as_of` 防止未来信息；`schema_version` 防止隐式破坏；`trace_id` 支持跨 Agent 和跨系统排障；`policy_decision_id` 证明为什么允许这次访问或写入。

## 4. 交易关键路径与第二大脑的“物理断开”

### 4.1 允许路径

```text
第二大脑 → ResearchProposal / EvidenceClaimDraft / OrderIntentDraft
                             │
                             ▼
交易研究或策略服务（可拒绝、可忽略、可重放）
                             │
                             ▼
RuleSnapshot → DataQualityState → Strategy / Portfolio → RiskDecision
                             │                                  │
                             └──────人工审批 + 受控 OMS───────────┘
                                                        │
                                                        ▼
                                                     Broker
```

### 4.2 禁止路径

```text
LLM / Agent / Memory / Vector search ─X→ Broker API
第二大脑数据库 ─X→ 订单表、持仓表、风控白名单
外部网页 / PDF / 消息 ─X→ 改变权限、执行命令、放宽风险
检索结果“很像” ─X→ 当成实时市场事实或自动交易信号
```

### 4.3 失败模式与默认动作

| 故障 | 对第二大脑 | 对交易平台 | 安全默认 |
|---|---|---|---|
| 检索服务不可用 | 无法提供辅助解释 | 已批准策略、规则和风险仍可运行 | 不增加新风险；未知时 `NO_TRADE`。 |
| 事件投影延迟/重复 | 显示“知识滞后/待同步” | 不影响账本与订单状态 | 用 `event_id` 幂等；回源交易账本。 |
| 第二大脑出现错误结论 | 作为候选/冲突记录 | 不可进入风控裁决 | 降级为研究材料，要求原始证据。 |
| 交易系统风险引擎故障 | 可解释故障 | 不发送新订单 | 拒单、仅减仓或停机，取决于预定风险政策。 |
| 网关鉴权/Schema 失败 | 请求被拒绝 | 不被旁路 | 不做“临时直连”。 |

## 5. “一个项目集”究竟要共用什么、不共用什么

### 5.1 必须共用：治理与语义，而不是数据库

| 共同资产 | 为什么需要共同 | 实现方式 |
|---|---|---|
| `SYSTEM_CHARTER`、风险章程、ADR | 避免两个 AI/两套文档给出冲突边界。 | 项目集根目录中的版本化文档；变更需审计。 |
| 身份、角色、任务、工件、公告栏、Unknown Registry | Codex、WorkBuddy、Maibot、GPT 需要同一任务事实。 | 统一控制面，事件化登记，不授予跨域写权限。 |
| Schema Registry 与契约测试 | 避免“叫同一个名字、字段却不同”。 | 共享 `contracts/`，每个产品各自实现和 CI 验证。 |
| 证据与来源语义 | 交易研究与第二大脑必须都能表明来源、时间、限制、冲突。 | `EvidenceClaim` + provenance + license policy。 |
| 可观测性 ID | 发生事故时能追溯人、AI、输入、版本、结果。 | `task_id / trace_id / run_id / artifact_id`。 |

### 5.2 必须分开：权限、数据权威与发布节奏

| 不能共享的东西 | 原因 | 做法 |
|---|---|---|
| 实盘密钥、券商 token、订单路由凭据 | 最小权限和爆炸半径控制。 | 只在独立执行进程/Secrets Broker 内使用。 |
| 订单、持仓、现金的写库权限 | 防止知识系统或 Agent 污染金融真相。 | 第二大脑仅获取经脱敏、延迟和审计允许的投影。 |
| 交易核心与知识系统的发布耦合 | 知识模型/索引更新频繁，不应迫使交易停机。 | 各自版本、各自回滚、联动层做兼容检查。 |
| 交易热路径中的远程调用 | 检索、图谱、LLM 有不确定延迟和失败模式。 | 热路径无第二大脑 RPC；只使用已批准本地快照。 |

### 5.3 代码组织：现在采用单仓库项目集，未来可拆仓库

在当前单人/小团队、本地雏形尚未建立 Git 基线的阶段，推荐建立**一个 Git 根仓库（monorepo）**，但不是一个不可分的应用：

```text
intelligent-investment-program/
├── PROJECT_CHARTER.md                 # 项目集总纲：唯一
├── AGENTS.md                          # 短小硬规则：唯一
├── contracts/                         # 唯一共享代码面：Schema / API / events
├── control-plane/                     # Task / Artifact / Capability / Policy / Audit
├── trading-platform/                  # 可独立 build/test/deploy
│   ├── services/ libs/ adapters/ tests/ ops/
├── second-brain/                      # 可独立 build/test/deploy
│   ├── services/ memory/ knowledge/ assets/ tests/ ops/
├── integration-gateway/               # 可独立部署；只做适配、鉴权、验证、审计
├── skills/                            # 版本化 Manifest；默认沙箱与只读
├── docs/adr/                          # ADR-001 到 ADR-n
└── .github-or-local-ci/               # 契约、回放、安全、恢复测试
```

拆成两个代码仓库的触发条件不是“感觉更企业级”，而是以下至少两项已被事实证明：独立团队/权限域、独立发布节奏、独立容量与 SLA、独立合规边界、契约稳定且有成熟 CI。即使拆仓库，`contracts` 也应由独立版本库/包管理发布，绝不复制粘贴。

## 6. 实施顺序：共同设计、交易优先、第二大脑早期只读

### Phase 0：项目集基线（先做，1–2 周）

目标不是写大功能，而是让以后不会走错。

- 建立 Git 根仓库、目录所有权、变更锁和公告栏；`F:\aidanao` 现非 Git 仓库时，不假装使用多 worktree 并发写入。
- 固化 `PROJECT_CHARTER`、权限矩阵、`NO_TRADE`、灾停原则、ADR-001（本蓝图的结构决定）。
- 建立 `Task / Capability / Artifact / Unknown Registry`；审计 WorkBuddy 实际接口、数据、券商、L2、硬件和权限。
- 发布 v0 Schema：`DatasetSnapshot`、`RuleSnapshot`、`Experiment`、`DecisionPacket`、`EvidenceClaim`、`ResearchProposal`、`CognitiveHandoff`。
- 第二大脑只读扫描本地雏形，建立 `LegacyAdapter`；绝不移动、覆盖原文件或直接索引生产密钥。

**退出门槛：** 每个系统谁能读/写/审批/停止可以回答；所有未知项有验证方法；全局状态为 `research_only / NO_TRADE`。

### Phase 1：交易最小可信闭环 + 第二大脑只读旁路（优先，2–6 周）

- 交易：沿用已存在的 CSV/JSON → 标准行情 → 特征 → 策略 → SMA 回测闭环，补全原始/规范/特征/信号分层、规则版本、点时测试和可回放 `DecisionPacket`。
- 第二大脑：入库系统总纲、规则来源、实验、失败记录、公告栏与资产目录；只能查询，不向交易库写入。
- 联动层：实现交易 `DecisionPacket` 的只读投影和 `EvidenceClaim` 的引用回链。

**验收：** 关闭第二大脑后交易回放结果不变；从第二大脑任一研究结论能回到原始文件、数据快照、规则版本和试验 ID。

### Phase 2：交易研究工厂 + 证据/经验闭环（6–12 周）

- 交易：事件驱动回测、成本/容量/规则约束、试验族登记、滚动样本外与复现。
- 第二大脑：主张—证据—反证图谱、经验模板、冲突检索、前瞻提醒、资产哈希与版本。
- 联动：交易结果以候选经验回写；必须含适用市场、样本期、成本、反例、失效条件和复核日期。

**验收：** 任何“有效经验”都能查到验证状态；一次复盘不能自动成为策略规则。回测过拟合风险需进入验收，金融研究对 PBO/CSCV 的讨论支持这样做。[论文](https://www.risk.net/journal-of-computational-finance/2471206/the-probability-of-backtest-overfitting)

### Phase 3：影子/纸面交易与受控提案（12–20 周）

- 交易：实时入口字段审计、主源/补源/降级源、独立风险、OMS、对账、告警、影子再到纸面。
- 第二大脑：可生成研究任务、情景草案和 `OrderIntentDraft`，但始终需要确定性服务重新计算。
- 联动：把风险拒绝、数据异常、成交偏差、事故回写为知识候选和维修工单。

**验收：** 任意草案都能被拒绝；不能因第二大脑的不可用、幻觉、延迟或注入而多出一笔订单。

### Phase 4：只在全部门禁通过后讨论受控实盘（20 周以后）

实盘不是“功能完成”的自然下一步。只有券商能力和程序化交易要求、数据许可、稳定性演练、影子/纸面指标、独立风控、人工审批、灾备与对账同时通过，才可提交单独的受控执行审批。上交所现行细则对证券程序化交易作了专门约束，实施状态和账户适用性必须在启用当日核验。[规则](https://www.sse.com.cn/lawandrules/sselawsrules2025/trade/universal/c/c_20250612_10781696.shtml)

## 7. 对“交易效果会不会更好”的严谨回答

### 7.1 不会自动更好

把项目放在同一项目集、让第二大脑连接交易系统，**不会自动增加 Alpha，也不应提高下单速度**。若把检索、LLM、图谱或 Agent 放进热路径，通常只会提高延迟、故障面、提示注入风险和解释不一致风险。

### 7.2 可能变好的，是研究与运营质量

在以下条件满足时，第二大脑可通过间接路径改善交易研发：

```text
更完整的证据/失败试验/规则记忆
        ↓
更少重复试错、更少遗漏约束、更快定位数据和执行问题
        ↓
更好的研究预注册、样本外验证和复盘质量
        ↓
只有经独立回测、成本、风险与影子验证后，才可能改善策略结果
```

这正是把“知识系统”定位为研究质量基础设施而非信号机器的原因。大量调参后漂亮回测会提高过拟合风险，相关研究要求记录试验次数和样本外表现；这也是第二大脑应保存失败实验而非只保存成功故事的理由。[Bailey 等的讨论](https://scholarworks.wmich.edu/math_pubs/40/)

### 7.3 必须做 A/B 评估，禁止凭感觉上线

| 假设 | 对照 | 指标 | 通过条件 |
|---|---|---|---|
| 第二大脑提高研究检索 | 传统文件检索 / 无第二大脑 | 正确来源命中率、检索时长、错误引用率、人工复核率 | 提升有统计和人工证据，且无敏感泄露。 |
| 第二大脑提高实验质量 | 无提示的同类研究任务 | PIT/泄漏失败率、复现率、PBO/试验族记录完整率、被拒绝的坏假设数 | 质量门禁改善，不以回测 Sharpe 单项判断。 |
| 第二大脑提高运行恢复 | 无上下文的故障演练 | MTTR、误操作、回滚成功率、审计完整度 | 故障处理更快且无越权。 |
| 第二大脑提高交易结果 | 固定策略基准、冻结版本 | 样本外净收益、回撤、容量、交易成本、稳定性切片 | 必须成本后、跨市场状态、影子/纸面验证；无结果即不宣称。 |

## 8. A 股专属要求如何改变集成设计

1. **规则和市场状态必须版本化。** A 股的 T+1、交易时段、涨跌幅、停复牌、公司行动、板块差异、申报与费用不能由第二大脑“凭常识回答”；每个研究和订单草案要引用 `RuleSnapshot` 与 `as_of`。
2. **程序化交易不是普通软件集成。** 上交所发布的细则将通过程序自动生成或下达交易指令的行为纳入程序化交易定义，并覆盖股票、债券、基金、存托凭证等。因此任何由 AI 产生的订单草案都必须被视为高风险输入而非自动授权。[上交所规则](https://www.sse.com.cn/lawandrules/sselawsrules2025/trade/universal/c/c_20250612_10781696.shtml)
3. **“主力意图”只能是可证伪的概率化机制。** 第二大脑可保存支持/反证、特征、样本期和预测失效；不得把盘口或历史记忆转写为具体操盘主体身份或确定结论。
4. **实时数据与历史知识须隔离。** 实时行情、账户、订单、L2 可能有许可和敏感限制；第二大脑拿到的是按权限裁剪的投影。历史文档永远不能覆盖实时主数据。
5. **规则变化需要前瞻记忆，但触发执行前仍要复核。** 第二大脑可在规则到期、数据合同更新、回测窗口结束时提醒；具体启用必须由 Rule Engine 从官方和券商证据重新确认。

### 9.2 你大概率知道但尚未完全说出的隐性要求（必须由本地证据确认）

| 隐性要求 | 为什么合理 | 验证动作 | 未验证时默认 |

| 希望过去的文档、代码、数据、聊天、图表可持续被引用 | 否则“第二大脑”会断片。 | 盘点路径、格式、所有者、许可、哈希与更新时间。 | 只读索引，不迁移/不删除。 |

| 希望知识能帮助决策但不偷走控制权 | 这才符合交易安全。 | 审计每个工具的读写权限与策略。 | 只读与候选区。 |
| 希望本地雏形不断升级而非推倒重来 | 已有资料和闭环有价值。 | `LegacyAdapter` 对每项资产生成映射报告。 | 不破坏原位置。 |

### 9.3 你可能还不熟悉、但可理解并需要做选择的事项

| 项目对象 | 白话解释 | 需要的决定 | 当前推荐 |
|---|---|---|---|
| Canonical Schema | 两套系统交换数据的“共同语言”。 | 谁维护版本、字段如何改。 | `contracts/` 独立评审；破坏性变更需版本升级。 |
| 事件投影 | 把交易结果的副本发给第二大脑，而非共用数据库。 | 延迟、脱敏和保留期。 | 先离线/批次投影，后按需准实时。 |
| Policy Engine | 机器化地判断这次读写是否允许。 | 权限角色与审批规则。 | 默认拒绝、按请求验证。 |
| 契约测试 | 自动证明两边仍说同一种“语言”。 | CI 的门禁。 | 每次 Schema / Adapter 变更必跑。 |

### 9.4 深层未知：无法由 AI 直接补齐，必须显式登记

| 未知项 | 不能假装知道的原因 | 验证/管理方法 | 安全默认 |
|---|---|---|---|
| 哪个策略未来持续有效 | 市场会变、历史统计不等于未来。 | 样本外、影子/纸面、漂移监控、容量与成本检验。 | 不交易/小规模受控。 |
| “主力”真实身份和未来意图 | 不可完全观测。 | 竞争假设、概率、反证、弃权。 | 不把推断当事实。 |
| 券商、数据商、L2 的确切权限与字段语义 | 合同、版本和账户条件可变。 | 字段级 Capability Card 与书面证据。 | `unverified`、不接生产。 |
| 第二大脑是否真的提升收益 | 可能提升研发，也可能制造噪声。 | 预注册 A/B、冻结基准、成本后评估。 | 仅研究辅助。 |
| 未来监管、网络和供应链故障 | 不能预知。 | 规则前瞻提醒、演练、备份、回滚、人工审批。 | 安全停机。 |

## 10. 多 AI 与技能联动：谁做什么，谁永远不能做什么

| 参与方 | 在项目集中的角色 | 可以调用 | 禁止 |
|---|---|---|---|
| 用户 | 方向、资金、风险预算、生产审批 | 审批面、报告、完整审计 | 被系统静默替代高风险决策。 |
| ChatGPT / GPT | 研究、总设计、检索解释、任务分解、验收 | 受限上下文、候选知识、任务创建 | 修改本地生产、读取密钥、下单。 |
| Codex | 代码、测试、回放、架构、修复 | 任务范围内的工作树、沙箱、只读资源 | 虚构验证、扩大权限、直接上线/下单。 |
| WorkBuddy | Windows 本地接口、部署、运行、监控 | 受控适配器和运维任务 | 以未证实 MCP 快照声称券商/L2 能力、绕过风控。 |
| Maibot | 日常记忆、情绪/反思、提醒、维护请求 | 认知网关的低权限工具 | 自动修复生产代码、部署、交易。 |
| 自定义技能 | 一个明确输入输出的小能力 | Manifest 声明的数据与权限 | 访问未声明目录/网络/密钥，直接产生生产订单。 |

每个技能均有 `SkillManifest`：输入/输出 Schema、版本、数据范围、网络/文件系统策略、资源限制、测试、血缘、回滚版本、权限状态。接入流程固定为：**Manifest → Schema/权限检查 → 单测/金样本/PIT → 沙箱回放 → 影子调用 → 人工批准注册**。

跨系统 Agent 安全要按请求而不是仅在启动时鉴权；OWASP 的近期建议也强调最小权限、人工确认与完整操作日志。[参考](https://cornucopia.owasp.org/cards/AAIK)

          └─ P1-03 第二大脑 LegacyAdapter（只读）

### 30 天成功标准

- 交易闭环在第二大脑关闭时仍输出相同的可重放结果；
- 第二大脑可检索到经授权的交易研究工件，并回链至原始来源；
- 没有 Agent、技能或知识库具备生产密钥或订单写权限；
- 每个已知/未知能力都能查到证据、负责人和下一验证动作；

## 13. 架构验收清单

### 独立性

- [ ] 停止第二大脑后，交易数据质量、规则、风控、OMS、对账与紧急停机仍可工作。
- [ ] 停止交易平台后，第二大脑仍可管理非交易知识与资产，并把交易域标记为快照过期。
- [ ] 两侧没有共享生产写库账号、共享实盘密钥或无版本 API。

### 联动性

- [ ] 每个跨域请求经过身份、能力、敏感性、政策、Schema、审计检查。
- [ ] 任何交易引用可回到 `as_of`、规则版本、数据快照和试验；任何知识主张可回到来源和限制。
- [ ] `OrderIntentDraft` 无法在没有新鲜 `RuleSnapshot`、数据质量、风险裁决和人工批准的情况下产生订单。

### 可扩展性与可靠性

- [ ] 模块化单体内部有领域边界、依赖规则与契约测试；不是“一个文件式单体”。
- [ ] 服务拆分以量化证据决定：独立故障域、负载、权限、团队、发布节奏，而非追求名词。
- [ ] 事件幂等、版本兼容、死信、重放、备份和恢复已演练。
- [ ] 有端到端 `trace_id`；复杂分布式问题可定位，而非依赖聊天记录猜测。

## 14. 最终建议与立即行动

**建议正式采用：一个企业项目集 + 两个独立产品 + 一个受控联动层；从第一天统一章程、契约和审计；实施上坚持“交易最小可信闭环优先，第二大脑只读旁路同步建设”。**

这比“先把交易全部做完再想第二大脑”少返工，也比“从一开始把 AI、知识库和交易揉成一个巨型运行程序”更安全、更低延迟、更容易验证。它不会神奇地提高交易收益，但能为 A 股研究、规则更新、复盘、协作、风险解释和长期积累提供可测量的帮助；任何收益改进必须经独立的成本后、样本外、影子/纸面验证。

**下一条应执行的任务：** `【Codex模式：项目计划模式】P0-01/P0-02：只读审计现有本地雏形与真实能力，建立 Git 基线方案、能力清单、Unknown Registry 和 ADR-001；不迁移、不接券商、不下单。`
