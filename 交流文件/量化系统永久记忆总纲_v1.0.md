# 0. 快速唤醒卡

只读这一节，也应能快速恢复项目方向。

## 0.1 总目标

构建一套面向A股的、可验证、可回测、可仿真、可审计、最终可有限自动执行的量化交易系统。

目标不是承诺赚钱，而是：

> 在真实A股规则、T+1、涨跌停、停牌、交易成本、滑点和执行限制下，寻找能够保持样本外正期望的策略。

## 0.2 四方角色

- 用户：项目所有者，负责目标、预算、实盘、资金、凭证和高风险审批。
- ChatGPT：研究与系统总设计、任务调度、知识整合、结果审计和独立验收。
- Codex：首席技术架构师、核心系统建设者、代码质量负责人。
- WorkBuddy：本地运行平台、真实数据接口工程师、实时运维和执行结果打包者。

## 0.3 当前优先级

1. 完成交易系统最小可靠闭环；
2. 建立实时L2、逐笔成交、逐笔委托和集合竞价数据能力；
3. 统一历史逐笔文件与实时接口；
4. 建设消息面量化、量价、筹码、订单流、板块联动和风险系统；
5. 严格回测；
6. 仿真和影子模式；
7. 用户批准后才讨论有限实盘。

## 0.4 当前核心接口方向

- 保留现有通达信MCP作为快照、盘口、分钟线、上下文和降级源。
- TdxQuant作为候选实时原始流，必须先做字段级PoC。
- 用户已有历史逐笔成交文件，需要统一回放Adapter。
- WorkBuddy负责真实字段和运行验证。
- Codex负责统一Schema、Adapter、事件总线、订单簿和测试。

## 0.5 A股关键约束

- 开盘集合竞价：9:15–9:20可撤单；9:20–9:25不可撤单。
- 普通A股股票实行T+1库存约束。
- ST最新规则当前按上下10%处理。
- 生产程序不依赖固定百分比，优先读取当日 `upper_limit_price` / `lower_limit_price`。
- 停牌仓位不能从组合收益和风险中消失。
- 未经用户批准，任何AI不得真实下单。

# 3. 记忆条目状态

每条知识必须标状态。

| 状态 | 含义 |
|---|---|
| invariant | 安全、法律、市场机制等硬约束 |
| verified | 已有可靠数据或正式规则验证 |
| partially_verified | 有证据，但适用范围有限 |
| hypothesis | 待回测假设 |
| engineering_reference | 工程参考，不代表策略有效 |
| deprecated | 已废弃，保留原因 |
| conflict | 存在来源或定义冲突 |
| pending_source | 用户或系统提出，但官方来源待补 |

未经验证的内容不得直接升级为交易铁律。

# 6. 已确认的重要规则

## 6.1 组织规则

- Codex负责架构和核心建设。
- WorkBuddy负责真实接口、本地运行和运维。
- ChatGPT负责研究设计和验收。
- 用户保留最终审批权。
- Codex与WorkBuddy不得同时修改同一文件。

## 6.2 数据规则

- 原始数据不可覆盖。
- raw、normalized、features、signals分层保存。
- 必须保存交易所时间、供应商时间和系统接收时间。
- 序列号优先于时间戳重建事件顺序。
- 历史数据和实时数据必须使用统一Schema。
- 没有逐笔委托时，不得声称计算真实撤单率。

## 6.3 研究规则

- 不允许未来数据泄漏。
- 不允许把当前成分股回填历史。
- 不允许单信号决策。
- `+1/0/-1`只作展示，原始数值、概率和证据必须保留。
- Wyckoff Phase是状态特征和研究框架，不是绝对前置条件。
- “主力、机构、吸筹、出货、诱多”等只能作为概率推断。
- 高准确率不等于可交易收益。

## 6.4 交易规则

- T+1必须进入仓位账本和回测。
- 卖出后交易可用资金、可转出资金和可卖股数必须分开。
- ST最新基线按上下10%处理。
- 运行时以真实涨跌停价字段优先。
- 停牌仓位继续计入组合风险。
- 未经用户批准不得真实下单。

# 7. 当前重点研究模块

## 7.1 集合竞价

核心不是简单判断高开低开，而是分析：

- 9:15–9:20低承诺探索期；
- 9:20–9:25高承诺形成期；
- 虚拟开盘价；
- 匹配量；
- 未匹配量；
- 撤单和订单保留；
- 板块联动；
- 消息预期差；
- 开盘后5分钟、30分钟验证。

## 7.2 实时L2与逐笔

目标字段：

- 逐笔成交；
- 逐笔委托；
- order_id；
- trade_id；
- sequence_no；
- channel_no；
- new/cancel/modify；
- 买卖盘口；
- 集合竞价；
- 断线恢复；
- 本地永久落盘。

## 7.3 消息面量化

核心链条：

```text
消息事实
→ 业务关系
→ 市场预期
→ 预期差
→ 提前消化
→ 价格/成交量/波动响应
→ 时间衰减
→ 剩余定价空间
```

## 7.4 量价和微观结构

重点：

- Delta；
- CVD；
- OFI；
- Queue Imbalance；
- Absorption；
- VWAP；

- 涨跌停后的价格发现。

# 10. 外部知识摄入流程

以后ChatGPT、Codex或WorkBuddy发现有价值的论文、规则、接口或交易经验时：

```text
发现
→ 来源分级
→ 提炼结论
→ 标记适用范围

→ Codex建设
→ WorkBuddy取数运行
```

每条资料必须记录：

- 适用市场；
- 所需数据；
- 验证方法；
- 当前状态；
- 接受或否决原因。

# 11. 永久记忆更新机制

## 11.1 什么时候更新

以下情况必须生成“记忆补丁”：

- 用户明确纠正规则；
- 新接口能力被验证；
- 回测结果改变旧结论；
- 组织角色改变；
- 新模块正式加入；
- 某条铁律被降级或废弃；
- 重大交易复盘产生可泛化经验。

## 11.2 谁负责什么

- ChatGPT：判断什么值得进入长期记忆，并生成补丁内容。
- WorkBuddy：把补丁写入本文件，更新版本和变更日志。
- Codex：涉及架构和代码时，更新对应技术文档和引用。
- 用户：批准重大规则、实盘风控和组织变化。

## 11.3 更新方式

不每次复制一份新总纲。

采用：

```text
一个主文件
+ 小型记忆补丁
+ 版本日志
```

补丁格式：

```markdown
## MEMORY PATCH
patch_id:
date:
reason:
add:
modify:
deprecate:
evidence:
affected_files:
approved_by:
```

# 12. ChatGPT永久记忆边界

ChatGPT可以在当前项目、项目文件和平台记忆中保留部分上下文，但不能把平台内部记忆当作唯一可靠数据库。

因此本文件是外部永久记忆的唯一基准：

> 平台记忆负责“想起来”，本文件负责“不会丢”。

新会话中，只要上传最新版总纲，ChatGPT就能快速恢复项目方向。
