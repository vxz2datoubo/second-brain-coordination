# SECURITY-AND-PERMISSION-BOUNDARIES — 安全与许可边界分析

> 基于公开资料的初步判断，最终以券商书面确认为准。

## 一、安全隔离评估 — 各券商路线

### 1. QMT/miniQMT (xtquant)

**安全优势**:
- miniQMT 通过本地 TCP 连接，行情与交易通过不同端口/submodule
- xtdata 模块可以**独立初始化**，不初始化 xttrader 即无法下单
- 外部 Python 进程 (PyCharm/VSCode) 通过 `pip install xtquant` 调用，**完全在外部运行**
- xtdata 不需要交易密码即可获取行情（需登录账号确认权限）
- `xtdata.connect()` 仅连接行情服务，`xttrader` 需单独创建
- 方法白名单可行：仅 `import xtdata`，不 `import xttrader`

**安全风险**:
- QMT 完整版内置下单功能，UI 可触发交易
- miniQMT 运行在券商认可的环境内
- xttrader 模块存在但可选择性不加载

**结论**: ✅ **行情与交易可分离**。外部 Python 仅引入 `xtdata` 即可获取行情，不引入 `xttrader` 则无下单能力。安全隔离**可行**。

### 2. PTrade (恒生)

**安全优势**:
- 云端策略托管在券商机房，物理隔离
- 策略通过 Web UI 管理，不直接暴露 SDK

**安全风险**:
- **内网环境**，无法 pip 安装第三方库
- 策略编辑器中同时存在行情和交易函数
- 无法在外部独立进程中运行（除非极少数支持外网的券商）
- 数据获取需要在 PTrade 内部运行策略

**结论**: ⚠️ **行情与交易在同一进程**，但策略逻辑由用户控制。安全隔离**部分可行**（可写纯行情策略不调用order函数），但不如miniQMT外部进程那样物理隔离。

### 3. 华泰PTrade

**额外风险**:
- 云端策略托管，用户对执行环境控制权低
- 无法安装自定义Python包
- 策略源码需上传到券商云端

**结论**: ⚠️ 同PTrade通用评估，但增加了数据上传隐私顾虑。

## 二、许可限制预判 (基于行业惯例，待券商确认)

| 许可项 | 行业预期 | 风险等级 |
|------|------|:---:|

| 终止后删除数据 | 需逐家确认 | 🟡 中 |

## 三、禁止边界 (铁律)

以下为本次任务绝对禁止的操作：

- ❌ 登录券商账户 (在用户明确授权前)
- ❌ 提交交易密码
- ❌ 调用交易API (order/cancel/account/position)
- ❌ 自动下单
- ❌ 访问持仓、资产或委托
- ❌ 支付费用
- ❌ 承诺入金或交易量
- ❌ 签署协议
- ❌ 下载未知来源程序
- ❌ 开通任何收费服务
